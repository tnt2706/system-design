1 Nền tảng System Design

Bộ "từ vựng" và tư duy định lượng — nền móng cho mọi quyết định.

"Làm sao đo một hệ thống là tốt? Khi user tăng 100 lần thì cái gì gãy trước?"

Scalability — Vertical vs Horizontal · mở rộng hệ thống thế nào▶
Là gì

Scalability là khả năng hệ thống xử lý được nhiều tải hơn (user, request, dữ liệu) bằng cách thêm tài nguyên. Có 2 hướng:

  • Vertical scaling (scale up): làm 1 máy mạnh hơn — thêm CPU, RAM, ổ cứng.
  • Horizontal scaling (scale out): thêm nhiều máy chạy song song, chia tải cho chúng.

Cách làmNâng cấp 1 máyThêm nhiều máy
Giới hạnCó trần phần cứngGần như vô hạn
DowntimeThường phải restartKhông cần
Độ phức tạpĐơn giảnCao (cần load balancer, state phân tán)
SPOFCó (1 máy chết là toàn bộ chết)Không (1 máy chết, máy khác gánh)
Chốt: Vertical đơn giản, dùng trước khi hệ thống còn nhỏ. Khi chạm trần phần cứng hoặc cần độ sẵn sàng cao → bắt buộc đi horizontal. Hệ thống lớn gần như luôn horizontal. Điều kiện để scale out dễ dàng: service phải stateless (xem bài Stateless).

Ví dụ

Một API server quá tải: đầu tiên tăng RAM/CPU (vertical). Khi 1 máy không đủ, đặt 3 máy sau 1 load balancer (horizontal). Vì máy nào cũng xử lý được request bất kỳ (stateless), thêm máy thứ 4, 5… rất dễ.

Vertical: nâng cấp 1 máy (có trần phần cứng). Horizontal: thêm nhiều máy sau LB (gần như vô hạn, không SPOF) — điều kiện là service stateless.
Latency vs Throughput · nhanh vs nhiều▶
Là gì
  • Latency (độ trễ): thời gian để xử lý xong một request. Đo bằng ms. "Một request mất bao lâu?"
  • Throughput (thông lượng): số request xử lý được trong một đơn vị thời gian (RPS/QPS). "Bao nhiêu request mỗi giây?"

Hai thứ này độc lập: một hệ thống có thể throughput cao nhưng latency cũng cao (xử lý theo lô lớn), hoặc latency thấp nhưng throughput thấp.

Vì sao quan trọng

Khi tối ưu phải biết đang tối ưu cái nào. Thêm máy (horizontal) thường tăng throughput nhưng không giảm latency của 1 request. Muốn giảm latency phải tối ưu đường đi của chính request đó (cache, index, ít network hop).

Latency = độ dài ống (1 request mất bao lâu). Throughput = độ rộng ống (bao nhiêu/giây). Thêm máy → ống rộng hơn (throughput ↑) nhưng độ dài ống không đổi (latency 1 request không giảm).
Đo bằng percentile, không dùng trung bình: Latency thật được đo bằng p50, p95, p99 (99% request nhanh hơn ngưỡng này). Trung bình che giấu các request chậm. Với hệ thống có nhiều bước, p99 của tổng thể bị kéo bởi p99 của từng bước → "tail latency amplification".

Là gì

Availability = tỉ lệ thời gian hệ thống hoạt động đúng. Diễn đạt bằng số "9":

  • SLO (Objective): mục tiêu nội bộ, ví dụ "99.9% request < 300ms".
  • SLA (Agreement): cam kết với khách hàng, có điều khoản phạt nếu vi phạm. SLA luôn lỏng hơn SLO một chút để có biên an toàn.

Load Balancing · chia tải nhiều máy▶
Là gì

Load Balancer (LB) đứng trước nhiều server, nhận request và phân phối cho server phía sau. Là thành phần cốt lõi để horizontal scaling và tăng availability.

L4 vs L7
  • L4 (transport): phân phối dựa trên IP/port, không nhìn nội dung. Nhanh, đơn giản.
  • L7 (application): hiểu HTTP — có thể route theo URL path, header, cookie. Linh hoạt hơn (ví dụ /api đi server A, /img đi server B), làm được SSL termination.
Thuật toán phân phối
  • Round Robin: lần lượt từng server.
  • Least Connections: gửi tới server đang rảnh nhất.
  • Consistent Hashing: cùng một key (ví dụ userId) luôn tới cùng server — quan trọng khi cần "sticky" hoặc cache cục bộ (xem bài Consistent Hashing).
LB chia request cho các server khoẻ. Health check phát hiện Server 3 chết → tự động ngừng gửi tới nó (tự phục hồi). LB cũng cần redundant để không thành SPOF.
Health check: LB liên tục ping server; server nào "chết" thì ngừng gửi request tới — đây là cơ chế tự phục hồi cơ bản nhất. Bản thân LB cũng cần redundant để không thành SPOF.
CAP Theorem & Consistency models · đánh đổi cốt lõi của hệ phân tán▶
Là gì

Trong một hệ thống dữ liệu phân tán, khi có Network Partition (P) — mạng giữa các node bị đứt — bạn chỉ được chọn một trong hai:

  • Consistency (C): mọi node trả về dữ liệu mới nhất giống nhau (hoặc báo lỗi).
  • Availability (A): mọi request đều nhận được phản hồi (nhưng có thể là dữ liệu cũ).

Vì mạng chắc chắn sẽ có lúc đứt, P là bắt buộc. Nên thực chất bạn chọn giữa CP hoặc AP khi sự cố mạng xảy ra.

Hệ quả thực tế
LoạiKhi mạng đứtVí dụ dùng cho
CPTừ chối/chặn để giữ đúng dữ liệuNgân hàng, đặt vé, tồn kho
APVẫn trả lời, chấp nhận dữ liệu cũFeed mạng xã hội, like count, giỏ hàng
Consistency models (mức độ "đúng" của dữ liệu)
  • Strong consistency: đọc luôn thấy ghi mới nhất. Đắt, chậm.
  • Eventual consistency: nếu ngừng ghi, sau một lúc mọi node sẽ đồng nhất. Nhanh, rẻ, nhưng có thể đọc dữ liệu cũ tạm thời.
  • Read-your-own-writes: bạn luôn thấy được thứ chính bạn vừa ghi (dù người khác có thể chưa thấy).
Khi mạng giữa 2 node đứt: Node2 chưa nhận được X=2. Đọc Node2 → hoặc từ chối để giữ đúng (CP), hoặc trả dữ liệu cũ để vẫn phục vụ (AP). Phải chọn một.
Lỗi tư duy phổ biến: CAP chỉ áp dụng lúc có partition. Bình thường (mạng ổn) bạn có cả C lẫn A. Mô hình PACELC bổ sung: kể cả lúc bình thường (E - Else) vẫn phải đánh đổi Latency (L) và Consistency (C).
Stateless vs Stateful · chìa khoá để scale▶
Là gì
  • Stateless: server không lưu thông tin về request trước. Mỗi request mang đủ dữ liệu (ví dụ token) để xử lý độc lập.
  • Stateful: server giữ trạng thái (ví dụ session trong RAM) giữa các request.
Vì sao quan trọng

Stateless là điều kiện để horizontal scaling dễ dàng: request có thể đến bất kỳ server nào, thêm/bớt server tuỳ ý, một server chết không mất dữ liệu user. Stateful buộc request của 1 user phải về đúng server giữ session ("sticky session") → khó scale, mất session khi server chết.

Stateful: session nằm trong RAM Srv2 → request phải "dính" về Srv2. Stateless: session đẩy ra Redis chung → request đến server nào cũng xử lý được → thêm/bớt máy tự do.
Pattern chuẩn: Đẩy state ra ngoài — lưu session vào Redis/DB dùng chung, hoặc dùng token tự chứa (JWT). Khi đó service trở nên stateless. State vẫn tồn tại, chỉ là không nằm trong app server.
Back-of-the-envelope estimation · ước lượng nhanh quy mô▶
Là gì

Kỹ năng ước lượng nhanh QPS, storage, bandwidth bằng phép tính thô — để biết hệ thống cần bao nhiêu máy, DB chịu nổi không, cần cache hay không. Đây là kỹ năng luôn bị hỏi trong phỏng vấn Architect.

Các con số nền cần thuộc
  • 1 ngày ≈ 86,400 giây (làm tròn ~10⁵).
  • Đọc từ RAM: ~100 ns · SSD: ~100 µs · Disk/seek: ~10 ms · Round-trip trong cùng datacenter: ~0.5 ms · Cross-region: ~50–150 ms.
  • 1 char ≈ 1 byte; 1 triệu = 10⁶; 1 tỉ = 10⁹.
Ví dụ mẫu — Twitter

300 triệu user, mỗi người đăng 2 tweet/ngày → 600 triệu tweet/ngày ÷ 86,400 ≈ ~7,000 writes/giây. Đọc thường gấp ~100 lần ghi → ~700,000 reads/giây. Mỗi tweet ~300 bytes → 600M × 300B ≈ 180 GB/ngày ≈ 65 TB/năm.

Mục đích không phải con số chính xác mà là cấp độ (order of magnitude): "đây là hệ read-heavy 700k QPS → DB đơn không kham nổi → cần cache + replica đọc". Kết luận đó mới là thứ Architect cần.

SQL vs NoSQL — chọn theo access pattern · quyết định nền tảng▶
Là gì
  • SQL (relational): dữ liệu có schema cố định, quan hệ rõ ràng, hỗ trợ JOIN và transaction ACID mạnh. VD: PostgreSQL, MySQL.
  • NoSQL: nhóm DB linh hoạt schema, tối ưu cho scale ngang và truy vấn theo pattern cụ thể. Gồm nhiều loại.

Indexing — B-Tree vs LSM-Tree · vì sao query nhanh/chậm▶
Là gì

Index là cấu trúc dữ liệu phụ giúp tìm hàng cần thiết mà không phải quét toàn bảng — đổi tốc độ đọc lấy chi phí ghi + bộ nhớ (mỗi lần ghi phải cập nhật index).

Hai họ index lớn
  • B-Tree (PostgreSQL, MySQL): cây cân bằng, tối ưu cho đọc và range query (WHERE age > 20). Ghi tại chỗ → đọc nhanh, ghi vừa phải.
  • LSM-Tree (Cassandra, RocksDB, LevelDB): ghi nối tiếp vào bộ nhớ rồi flush xuống đĩa theo lô, gộp định kỳ (compaction). Ghi cực nhanh, đọc có thể chậm hơn (phải xem nhiều tầng).
Chốt: Hệ read-heavy → B-Tree. Hệ write-heavy (log, metrics, IoT) → LSM-Tree. Đây chính là lý do Cassandra giỏi ghi còn Postgres giỏi truy vấn linh hoạt.
B-Tree: cây cân bằng, đọc & range nhanh (Postgres/MySQL). LSM: ghi vào RAM rồi flush nối tiếp xuống nhiều SSTable, gộp định kỳ → ghi cực nhanh (Cassandra).
Lưu ý: Đừng index mọi cột — mỗi index làm chậm ghi và tốn dung lượng. Index sai (không khớp truy vấn) thì vô dụng. Hiểu composite index và quy tắc "leftmost prefix".
Replication · nhân bản dữ liệu▶
Là gì

Sao chép cùng một dữ liệu sang nhiều node. Mục đích: tăng availability (1 node chết vẫn còn bản khác), tăng throughput đọc (chia tải đọc cho nhiều replica), giảm latency (đặt replica gần user).

Các mô hình
  • Leader-Follower (single-leader): mọi ghi vào leader, leader sao chép sang follower; đọc có thể từ follower. Phổ biến nhất. Đơn giản, nhưng leader là điểm nghẽn ghi.
  • Multi-Leader: nhiều node nhận ghi (ví dụ multi-region). Throughput ghi cao nhưng phải xử lý conflict khi 2 nơi cùng sửa.
  • Leaderless (Dynamo, Cassandra): client ghi/đọc nhiều node, dùng quorum.
Sync vs Async & Replication Lag

Synchronous: leader chờ follower xác nhận → an toàn nhưng chậm. Asynchronous: leader trả lời ngay → nhanh nhưng follower trễ một chút (replication lag). Lag gây hiện tượng "vừa ghi xong, đọc lại không thấy" → cần read-your-own-writes (đọc từ leader cho dữ liệu vừa ghi).

Mọi GHI vào Leader → sao chép sang Follower (có độ trễ = replication lag). ĐỌC chia tải cho Follower → tăng throughput đọc. Nếu async, follower có thể trễ vài chục ms.
Sharding / Partitioning · chia nhỏ dữ liệu▶
Là gì

Khi dữ liệu/tải ghi quá lớn cho 1 node, chia dữ liệu thành nhiều phần (shard) đặt trên các node khác nhau. Khác replication (nhân bản toàn bộ) — sharding mỗi node giữ một phần khác nhau. Đây là cách duy nhất để scale ghi vượt giới hạn 1 máy.

Chiến lược chia
  • Range-based: chia theo khoảng (A–M, N–Z). Range query dễ, nhưng dễ hotspot (một khoảng nóng hơn).
  • Hash-based: hash key rồi chia. Phân bố đều, nhưng range query khó.
  • Consistent hashing: giảm số dữ liệu phải di chuyển khi thêm/bớt node (xem bài riêng).
Vấn đề phải giải
  • Hotspot: một shard quá tải (ví dụ shard chứa celebrity). Giải: chọn shard key phân tán đều.
  • Rebalancing: thêm node phải di chuyển dữ liệu — tốn kém, dễ gây gián đoạn.
  • Cross-shard query/JOIN: truy vấn nhiều shard rất đắt → cố thiết kế để query chỉ chạm 1 shard.
Khác replication (nhân bản toàn bộ): sharding chia dữ liệu thành phần khác nhau trên các máy. Là cách duy nhất scale GHI vượt 1 máy. Query chạm đúng 1 shard thì nhanh; chạm nhiều shard rất đắt.
Chốt: Chọn shard key là quyết định khó sửa nhất. Key tốt: phân bố đều + đa số query chỉ cần 1 shard. Sharding là "vũ khí cuối" — dùng khi replication + cache không còn đủ, vì nó tăng độ phức tạp rất nhiều.
Transactions, ACID & Isolation Levels · tính đúng đắn của dữ liệu▶
ACID là gì
  • Atomicity: toàn bộ transaction thành công hết hoặc rollback hết.
  • Consistency: dữ liệu luôn thoả ràng buộc (constraint) trước và sau.
  • Isolation: các transaction chạy song song không "dẫm chân" nhau.
  • Durability: đã commit thì không mất kể cả mất điện.
Isolation Levels & anomaly

Isolation càng cao càng đúng nhưng càng chậm. Mỗi cấp ngăn được một loại lỗi:

LevelNgăn được
Read Uncommitted(yếu nhất) — có dirty read
Read Committedngăn dirty read
Repeatable Readngăn thêm non-repeatable read
Serializablengăn cả phantom read — như chạy tuần tự, an toàn nhất, chậm nhất

Anomaly: Dirty read = đọc dữ liệu chưa commit; Non-repeatable read = đọc cùng hàng 2 lần ra khác nhau; Phantom = cùng query ra thêm/bớt số hàng.

Chốt: Đa số DB mặc định Read Committed. Nâng isolation chỉ khi nghiệp vụ cần (ví dụ chuyển tiền). Đây là trade-off đúng-đắn vs hiệu năng — Architect phải chọn có ý thức.
Caching — chiến lược & cạm bẫy · tăng tốc đọc▶
Là gì & vì sao

Cache lưu dữ liệu hay dùng ở nơi truy cập nhanh (RAM) để tránh đụng DB/tính toán lại. Giảm latency, giảm tải DB. Nhưng cache đưa vào bài toán đồng bộ — dữ liệu cache có thể cũ.

Các chiến lược
  • Cache-aside (lazy): app đọc cache trước; miss thì đọc DB rồi ghi vào cache. Phổ biến nhất. Đơn giản, nhưng lần miss đầu chậm.
  • Write-through: ghi vào cache và DB cùng lúc. Cache luôn mới, nhưng ghi chậm hơn.
  • Write-back (write-behind): ghi vào cache trước, flush xuống DB sau. Ghi cực nhanh, nhưng rủi ro mất dữ liệu nếu cache chết.
Eviction & tầng cache

Khi cache đầy phải đuổi bớt: LRU (ít dùng gần đây nhất), LFU (ít dùng nhất), TTL (hết hạn theo thời gian). Cache có nhiều tầng: client → CDN → app-level (Redis) → DB cache.

3 cạm bẫy kinh điển:
• Cache invalidation ("một trong hai việc khó nhất CS"): làm sao cache cũ bị xoá/cập nhật đúng lúc.
• Thundering herd / Cache stampede: một key hot hết hạn, hàng ngàn request cùng lúc đập vào DB. Giải: lock/single-flight, hoặc làm mới chủ động trước khi hết hạn.
• Cache penetration: query key không tồn tại liên tục → luôn miss → đập DB. Giải: cache cả giá trị "null", hoặc Bloom filter.
Cache-aside: App hỏi cache trước; HIT → trả ngay (nhanh, không đụng DB); MISS → đọc DB rồi ghi vào cache cho lần sau. Phần lớn đọc chỉ chạm RAM.
CDN & Content Delivery · đưa dữ liệu gần user▶
Là gì

CDN = mạng máy chủ đặt rải khắp thế giới, cache nội dung tĩnh (ảnh, JS, CSS, video) gần user. User ở Việt Nam tải ảnh từ server Singapore thay vì Mỹ → nhanh hơn nhiều và giảm tải cho origin server.

Cách hoạt động

Lần đầu CDN edge miss → lấy từ origin và cache lại. Lần sau phục vụ trực tiếp từ edge. Kiểm soát qua HTTP header Cache-Control, ETag. Để cập nhật file đã đổi mà vẫn cache lâu → dùng versioned URL (app.v2.js) hoặc content hash.

Chốt: CDN là tầng cache rẻ và hiệu quả nhất cho nội dung tĩnh và read-heavy. Bất kỳ thiết kế hệ thống có asset tĩnh hoặc phục vụ toàn cầu nào cũng nên có CDN ở phía trước.
Consistent Hashing · chia & rebalance ít đau nhất▶
Vấn đề với hash thường

Cách ngây thơ để chọn node giữ key: node = hash(key) % N (N = số node). Nhưng khi thêm/bớt một node, N đổi → gần như mọi key bị ánh xạ lại → phải di chuyển ~toàn bộ dữ liệu/cache. Thảm hoạ ở quy mô lớn.

Ý tưởng Consistent Hashing

Hình dung một vòng tròn (0 → 2³²). Hash cả node lẫn key lên vòng này. Mỗi key thuộc về node kế tiếp theo chiều kim đồng hồ. Khi thêm/bớt một node, chỉ các key giữa node đó và node trước phải di chuyển — phần còn lại không đổi.

Key đi theo chiều kim đồng hồ tới node gần nhất. Thêm/bớt node chỉ ảnh hưởng một cung nhỏ.
Virtual nodes: Để tránh phân bố lệch (một node ôm cung quá lớn), mỗi node vật lý được đặt ở nhiều điểm ảo trên vòng → tải đều hơn. Đây là cơ chế cốt lõi của Cassandra, DynamoDB, và các LB/cache phân tán. Bạn sẽ tự cài nó trong dự án Mini KV Store #3.
Probabilistic structures: Bloom Filter & HyperLogLog · đổi độ chính xác lấy bộ nhớ▶
Bloom Filter — "chắc chắn không có" hoặc "có thể có"

Cấu trúc cực tiết kiệm bộ nhớ để kiểm tra "phần tử này có thể đã tồn tại chưa?". Đặc tính: không bao giờ false negative (nếu nói "không có" thì chắc chắn không có), nhưng có thể false positive (nói "có thể có" mà thực ra không). Dùng nhiều hàm hash đánh dấu bit trong một mảng bit.

Ứng dụng: tránh truy vấn DB/đĩa vô ích — "username này đã dùng chưa?", chống cache penetration ("key này chắc chắn không tồn tại → khỏi đập DB"), LSM-tree dùng để bỏ qua SSTable không chứa key.

HyperLogLog — đếm số phần tử duy nhất gần đúng

Đếm "có bao nhiêu user khác nhau truy cập?" với hàng tỉ phần tử mà chỉ tốn vài KB (thay vì lưu toàn bộ set). Sai số ~1–2% nhưng bộ nhớ gần như cố định.

Ứng dụng: đếm unique visitors, unique search query, cardinality trong analytics (Redis có sẵn lệnh PFADD/PFCOUNT).

Bloom filter: hash phần tử bằng nhiều hàm, bật các bit tương ứng. Tra cứu chỉ xem các bit → tốn rất ít bộ nhớ. Không bao giờ false negative, có thể false positive.
Bài học Architect: Ở quy mô lớn, "đúng tuyệt đối" thường không đáng giá — chấp nhận sai số nhỏ để tiết kiệm bộ nhớ/tốc độ khổng lồ là một đánh đổi thông minh. Biết khi nào dùng cấu trúc xác suất là dấu hiệu tư duy chín.
Blob / Object Storage (S3-style) · lưu file lớn đúng cách▶
Là gì

Object storage (Amazon S3, GCS, MinIO) lưu file lớn — ảnh, video, backup, log — dưới dạng "object" có key, truy cập qua API HTTP. Khác biệt: scale gần như vô hạn, rẻ, bền (durability rất cao), nhưng không phải DB (không query phức tạp, không transaction).

Nguyên tắc thiết kế quan trọng
  • Đừng nhét file lớn vào DB quan hệ. Lưu file trên object storage, chỉ lưu URL/metadata trong DB. DB nhẹ, file scale riêng.
  • Presigned URL: cho client upload/download trực tiếp với storage (không qua app server) → app server không thành nút thắt băng thông.
  • Đặt CDN trước object storage cho nội dung tĩnh đọc nhiều.
Mẫu chuẩn: file lớn upload thẳng lên Object Storage (presigned, không qua App), DB chỉ giữ metadata + URL, CDN phục vụ file gần user. App không thành nút thắt băng thông.
Chốt: Hầu hết hệ thống có ảnh/video (mạng xã hội, e-commerce, streaming) đều theo mẫu: metadata trong DB + file trong object storage + CDN phía trước. Nhớ mẫu này là gần như xong phần "lưu trữ media" trong mọi bài thiết kế.
Concurrency Control & Distributed Lock · chống race condition▶
Vấn đề: Race Condition

Hai request cùng đọc số dư 100, cùng trừ 60 → cả hai thấy "đủ" → trừ 2 lần thành -20. Khi nhiều tiến trình cùng sửa một dữ liệu, phải có cơ chế kiểm soát.

Pessimistic vs Optimistic Locking
Pessimistic: khoá dòng trước khi sửa (an toàn, ít song song) — tranh chấp cao. Optimistic: không khoá, kiểm version lúc ghi, đụng thì retry (song song cao) — tranh chấp thấp.
Distributed Lock (khoá xuyên nhiều server)

Khi lock không thể nằm trong 1 DB (nhiều service/instance), cần khoá phân tán — thường dùng Redis: SET key value NX EX 30 (chỉ set nếu chưa tồn tại, kèm TTL). Ai set được thì giữ khoá; TTL đảm bảo khoá tự nhả nếu holder chết (tránh deadlock vĩnh viễn).

Cạm bẫy distributed lock: nếu holder xử lý lâu hơn TTL, khoá hết hạn trong khi nó vẫn chạy → 2 nơi cùng vào critical section. Redlock (dùng nhiều Redis độc lập) tăng an toàn nhưng vẫn gây tranh cãi. Lời khuyên: hạn chế phụ thuộc distributed lock; ưu tiên thiết kế idempotent hoặc atomic operation (như Redis DECR trong bài Flash-sale) thay vì khoá khi có thể.
Chốt: Tranh chấp thấp → optimistic (rẻ, không khoá). Tranh chấp cao trên một dòng → pessimistic. Xuyên service → distributed lock có TTL, nhưng cố tránh bằng atomic op + idempotency.
Schema Migration không downtime (Expand–Contract) · đổi DB khi đang chạy▶
Vấn đề

DB đang phục vụ khách 24/7, cần đổi schema (đổi tên cột, tách bảng, đổi kiểu) mà không downtime, không mất dữ liệu, rollback được. Đổi trực tiếp (rename cột) sẽ làm code cũ đang chạy hỏng ngay. Đây là kỹ năng vận hành Architect bắt buộc nắm.

Expand–Contract (Parallel Change) — 3 pha
Expand: thêm cấu trúc mới, ghi cả cũ lẫn mới. Migrate: backfill dữ liệu cũ + chuyển code đọc sang mới. Contract: bỏ cái cũ. Mỗi pha tương thích ngược nên không bao giờ downtime.
Ví dụ: đổi tên cột name → full_name
  • Expand: thêm cột full_name; app ghi vào cả hai.
  • Migrate: backfill full_name từ name cho dữ liệu cũ; chuyển app đọc full_name.
  • Contract: ngừng ghi name; sau khi yên tâm → drop cột name.
Nguyên tắc: mọi bước phải tương thích ngược để code phiên bản cũ & mới chạy song song trong lúc rolling deploy. Không bao giờ đổi phá vỡ trong một bước. Đây cũng chính là tư duy của Strangler Fig (GĐ6) áp cho dữ liệu.

Networking nền tảng — điều gì xảy ra khi gõ một URL · DNS, TCP, TLS, HTTP▶
Vòng đời một request (phải hiểu rõ)

Trước khi nói chuyện service-to-service, phải hiểu một request đi qua đâu. Mỗi bước thêm latency & là một điểm có thể hỏng/tối ưu:

Gõ URL → ① phân giải DNS ra IP → ② bắt tay TCP (SYN/SYN-ACK/ACK) → ③ bắt tay TLS (mã hoá) → ④ gửi HTTP request → ⑤ nhận response. Mỗi round-trip tốn latency (vì sao keep-alive & CDN gần user lại quan trọng).
TCP vs UDP
  • TCP: tin cậy, đúng thứ tự, có handshake & retransmit. Dùng cho web/API/đa số. Đánh đổi: chậm hơn (overhead).
  • UDP: "bắn và quên", không đảm bảo tới/đúng thứ tự, nhưng nhanh & nhẹ. Dùng cho video call, game, DNS, streaming real-time (mất vài gói chấp nhận được).
HTTP/1.1 vs HTTP/2 vs HTTP/3
Phiên bảnĐiểm chính
HTTP/1.1Một request/response mỗi kết nối tại một thời điểm → head-of-line blocking. Phải mở nhiều kết nối.
HTTP/2Multiplexing: nhiều request song song trên 1 kết nối, header nén. Nhanh hơn nhiều. (gRPC chạy trên đây.)
HTTP/3Chạy trên QUIC (UDP) → loại bỏ head-of-line blocking ở tầng transport, kết nối lại nhanh khi đổi mạng (mobile). Tương lai.
Real-time: WebSocket vs SSE vs Long-polling
  • Long-polling: client hỏi liên tục, server giữ tới khi có dữ liệu. Đơn giản, tốn kém, trễ.
  • SSE (Server-Sent Events): server đẩy 1 chiều (server→client) qua HTTP. Hợp feed/notification.
  • WebSocket: kênh 2 chiều mở liên tục. Hợp chat, game, collab real-time (xem bài Chat).
Vì sao Architect cần: nhiều quyết định (chọn gRPC vs REST, vì sao cần CDN/keep-alive, dùng WebSocket hay SSE, TCP hay UDP) đều bắt nguồn từ hiểu tầng mạng. Latency của một hệ thống phân tán phần lớn là latency mạng.
REST vs gRPC vs GraphQL · chọn kiểu API▶
Tóm tắt từng loại
  • REST: dùng HTTP + JSON, resource-based (GET /users/1). Đơn giản, phổ biến, dễ cache, ai cũng hiểu. Nhược: dễ over-fetch/under-fetch (lấy thừa hoặc thiếu field).
  • gRPC: dùng HTTP/2 + Protobuf (nhị phân). Rất nhanh, contract chặt, hỗ trợ streaming 2 chiều. Lý tưởng cho giao tiếp service-to-service nội bộ. Nhược: browser không gọi trực tiếp dễ, khó debug bằng mắt.
  • GraphQL: client tự chọn đúng field cần qua một endpoint. Giải quyết over/under-fetch, hợp cho mobile/UI phức tạp. Nhược: cache khó hơn, dễ query nặng, phức tạp phía server.

Synchronous vs Asynchronous & Message Queue · tách rời service▶
Sync vs Async
  • Synchronous: A gọi B và chờ phản hồi. Đơn giản, nhưng A phụ thuộc B — B chậm/chết thì A cũng chậm/chết (coupling chặt).
  • Asynchronous: A gửi message vào hàng đợi rồi đi tiếp; B xử lý sau. A và B decoupled.
Message Queue đem lại gì
  • Decoupling: producer và consumer không cần biết nhau, không cần online cùng lúc.
  • Buffering / load leveling: traffic dồn dập được "hấp thụ" vào queue, consumer xử lý theo nhịp của nó → không sập.
  • Resilience: consumer chết, message vẫn nằm trong queue, xử lý lại sau.

Ví dụ điển hình: gửi email/thông báo, xử lý ảnh/video, xử lý đơn hàng nền. RabbitMQ (message broker truyền thống) hợp task queue; Kafka (xem bài sau) hợp event stream lượng lớn.

Cái giá của async: phức tạp hơn (phải theo dõi message, xử lý lỗi, Dead Letter Queue cho message thất bại), khó debug, kết quả không tức thì. Đừng async hoá thứ vốn cần phản hồi ngay lập tức.
Sync: A gọi B và chờ → coupling chặt, B chậm kéo A chậm. Async qua Queue: A gửi rồi đi tiếp, B xử lý theo nhịp → tách rời, hấp thụ burst, chịu lỗi tốt hơn.
Event Streaming (Kafka) & Pub/Sub · luồng sự kiện lớn▶
Khác biệt với Message Queue thường

Queue truyền thống: message bị xoá sau khi consumer xử lý. Kafka: sự kiện được lưu lại (log bền vững) trong khoảng thời gian cấu hình; nhiều consumer group đọc độc lập, có thể "tua lại" (replay) từ đầu. Kafka là một commit log phân tán.

Khái niệm cốt lõi
  • Topic: dòng sự kiện theo chủ đề. Partition: topic chia nhiều phần để song song hoá (và giữ thứ tự trong từng partition).
  • Producer ghi, Consumer group đọc; offset đánh dấu đã đọc tới đâu.
  • Pub/Sub: mô hình 1 sự kiện → nhiều subscriber nhận. Kafka làm pub/sub ở quy mô rất lớn.
Khi nào dùng: log/metrics/clickstream lượng cực lớn, event-driven architecture, cần nhiều hệ tiêu thụ cùng một luồng sự kiện, cần replay. Đừng dùng Kafka cho task queue đơn giản — nó nặng và phức tạp hơn cần thiết.
Queue thường xoá message sau khi xử lý. Kafka giữ message trong log theo offset; nhiều consumer group đọc độc lập với offset riêng và có thể tua lại — đặc trưng cốt lõi.
Idempotency · làm 2 lần như 1 lần▶
Là gì & vì sao sống còn

Một thao tác idempotent nghĩa là thực hiện nhiều lần cho kết quả giống như một lần. Trong hệ phân tán, request chắc chắn sẽ bị gửi lại (timeout, retry, network glitch) → nếu thao tác không idempotent, bạn sẽ trừ tiền 2 lần, tạo đơn 2 lần.

Cách làm
  • Client gửi kèm Idempotency Key (UUID) cho mỗi thao tác.
  • Server lưu key đã xử lý; nếu thấy key trùng → trả lại kết quả cũ, không thực thi lại.
  • Một số thao tác idempotent tự nhiên: GET, PUT (set giá trị), xoá theo id. POST tạo mới thường không idempotent → cần key.
Chốt: Bất kỳ thao tác có side-effect tiền bạc/đơn hàng nào trong hệ phân tán đều phải idempotent. Đây là một trong những thứ phỏng vấn Architect rất hay đào sâu.
Client gửi kèm idempotency key. Lần 1: thực thi & lưu key. Lần retry cùng key: server trả lại kết quả cũ, không thực thi lại → không double-charge.
Distributed Transactions, 2PC & Saga · giao dịch xuyên service▶
Vấn đề

Trong microservices, một nghiệp vụ (đặt hàng = trừ kho + thanh toán + tạo đơn) trải trên nhiều service/DB khác nhau. Không còn transaction ACID đơn lẻ bao trùm tất cả. Làm sao đảm bảo "tất cả hoặc không gì cả"?

Hai cách
  • 2PC (Two-Phase Commit): một coordinator hỏi tất cả "sẵn sàng commit?" (phase 1), nếu tất cả đồng ý thì ra lệnh commit (phase 2). Đảm bảo mạnh nhưng: chậm, khoá tài nguyên lâu, coordinator chết là kẹt toàn bộ → thường tránh trong hệ thống quy mô lớn.
  • Saga (khuyến nghị): chia thành chuỗi transaction cục bộ, mỗi bước có một compensating action (hành động bù trừ) để hoàn tác nếu bước sau thất bại. Ví dụ thanh toán lỗi → "hoàn kho" + "huỷ đơn".
Hai kiểu Saga
  • Choreography: các service phản ứng với event của nhau, không có điều phối trung tâm. Phi tập trung, nhưng khó theo dõi luồng.
  • Orchestration: một orchestrator điều khiển từng bước. Dễ theo dõi/debug, nhưng orchestrator là điểm tập trung.
Saga: mỗi bước thành công đi tiếp; một bước lỗi → chạy hành động bù trừ (compensation) cho các bước đã làm.
Chốt: Saga đánh đổi: không còn atomicity tức thời, mà là eventual consistency + bù trừ. Đây là cái giá của microservices. Hiểu rõ Saga là dấu hiệu tư duy Architect chín.
Consensus, Raft & Leader Election · nhiều node đồng thuận▶
Vấn đề

Nhiều node cần thống nhất một giá trị/quyết định dù mạng có thể trễ, mất gói, node chết. Ví dụ: ai là leader? thứ tự các thao tác là gì? Đây là bài toán consensus.

Raft (dễ hiểu hơn Paxos)
  • Leader election: các node bầu một leader. Nếu leader im lặng quá lâu, node khác trở thành ứng viên và xin phiếu; ai được đa số (majority/quorum) thì làm leader mới.
  • Log replication: leader nhận mọi ghi, ghi vào log của mình rồi sao chép sang follower. Khi đa số đã ghi → coi như committed.
  • Majority quorum: mọi quyết định cần > nửa số node đồng ý → đảm bảo không bao giờ có 2 leader hợp lệ cùng lúc (tránh split-brain).
Ở đâu dùng: etcd, Consul, ZooKeeper, Kafka controller, các DB phân tán — tất cả dùng consensus để bầu leader và giữ dữ liệu nhất quán. Bạn không cần tự cài Raft, nhưng phải hiểu ý tưởng để biết vì sao các hệ này cần số node lẻ (3, 5) và chịu được mất (n-1)/2 node.
Raft: 1 leader nhận ghi, sao chép log sang follower; chỉ commit khi ĐA SỐ xác nhận. Vì cần majority nên hệ dùng số node lẻ (3,5) và chịu được mất (n-1)/2 node.
Event Sourcing & CQRS · lưu sự kiện thay vì trạng thái▶
Event Sourcing

Thay vì lưu trạng thái hiện tại ("số dư = 100"), lưu toàn bộ chuỗi sự kiện ("nạp 150", "rút 50"). Trạng thái hiện tại = phát lại các event. Lợi: có audit log đầy đủ, tua lại lịch sử, debug "tại sao tới trạng thái này". Hại: phức tạp, query trạng thái hiện tại tốn công.

CQRS (Command Query Responsibility Segregation)

Tách ghi (command) và đọc (query) thành 2 mô hình riêng, thường 2 store riêng. Ghi tối ưu cho đúng đắn, đọc tối ưu cho tốc độ (ví dụ materialized view). Hợp khi đọc và ghi có pattern rất khác nhau.

Cảnh báo over-engineering: Event Sourcing + CQRS rất mạnh nhưng phức tạp cao và gây eventual consistency giữa ghi và đọc. Chỉ dùng khi thực sự cần audit/replay/đọc-ghi lệch lớn (tài chính, đặt vé, hệ thống kế toán). Với CRUD bình thường thì đây là dùng dao mổ trâu giết gà.
Event Sourcing: lưu mọi sự kiện, trạng thái = replay lại (có audit, tua lại). CQRS: tách đường GHI (đúng đắn) và ĐỌC (tối ưu tốc độ) thành mô hình/store riêng.
WAL & Change Data Capture (CDC) · đồng bộ dữ liệu đáng tin▶
Write-Ahead Log (WAL)

Trước khi áp thay đổi vào dữ liệu, DB ghi thay đổi đó vào một log nối tiếp (append-only) trước. Nhờ đó nếu sập giữa chừng, khi khởi động lại DB phát lại WAL để phục hồi → đảm bảo Durability (chữ D trong ACID). WAL cũng là nguồn cho replication.

Change Data Capture (CDC)

Kỹ thuật "nghe" WAL của DB để bắt mọi thay đổi (insert/update/delete) và phát thành luồng event. Giúp đồng bộ dữ liệu sang hệ khác (search index, cache, data warehouse, service khác) mà không phải sửa code ứng dụng.

Vì sao quan trọng: CDC giải bài toán "dual write" kinh điển — khi bạn ghi DB và publish event, hai thao tác có thể lệch nhau lúc lỗi. Thay vào đó, ghi DB rồi để CDC sinh event từ chính WAL → nguồn sự thật duy nhất, không bao giờ lệch. Công cụ: Debezium + Kafka. Đây là nền của nhiều kiến trúc event-driven hiện đại và liên quan chặt tới Outbox pattern.
CDC đọc WAL của DB → phát mọi thay đổi lên Kafka → đồng bộ sang search index, cache, warehouse... mà không sửa code app. Giải bài "dual write" (nguồn sự thật duy nhất = WAL).
Batch vs Stream Processing (Lambda / Kappa) · xử lý dữ liệu lớn▶
Hai cách xử lý dữ liệu quy mô lớn
  • Batch processing: gom dữ liệu rồi xử lý theo lô lớn định kỳ (vd báo cáo cuối ngày). Thông lượng cao, độ trễ cao (kết quả không tức thì). Công cụ: Spark, Hadoop/MapReduce.
  • Stream processing: xử lý từng event ngay khi nó đến, gần real-time (vd phát hiện gian lận, dashboard live). Độ trễ thấp. Công cụ: Kafka Streams, Flink.
Hai kiến trúc tổng hợp
  • Lambda Architecture: chạy cả batch (chính xác, chậm) lẫn stream (nhanh, gần đúng) song song rồi hợp kết quả. Mạnh nhưng phải bảo trì 2 codebase.
  • Kappa Architecture: chỉ dùng một luồng stream cho mọi thứ; cần tính lại thì replay lại stream từ đầu. Đơn giản hơn, đang được ưa chuộng.
Áp dụng: Analytics/báo cáo định kỳ → batch. Cảnh báo/đề xuất/giám sát real-time → stream. Hiểu trục latency vs throughput vs độ chính xác để chọn — đây cũng là một dạng trade-off Architect.
Batch: gom rồi xử lý lô (chính xác, chậm). Stream: xử lý từng event tức thì (nhanh, gần đúng). Lambda kết hợp cả hai; Kappa chỉ dùng stream.
API Design chuyên sâu — contract bền vững · pagination, versioning, error▶

API là contract mà nhiều bên phụ thuộc vào trong nhiều năm — thiết kế tệ thì sửa rất đau. Đây là các quyết định Architect cần chuẩn hoá:

1. Resource naming & HTTP đúng nghĩa
  • Danh từ số nhiều, theo tài nguyên: GET /users/123/orders (không phải /getUserOrders).
  • Method đúng: GET (đọc, an toàn), POST (tạo), PUT (thay thế, idempotent), PATCH (sửa một phần), DELETE.
  • Status code đúng: 200/201/204, 400 (client sai), 401/403 (auth), 404, 409 (conflict), 422, 429 (rate limit), 5xx (server).
2. Pagination — offset vs cursor
Offset pagination chậm dần & dễ lệch ở dữ liệu thay đổi. Cursor pagination ("lấy N cái sau cursor X") nhanh ổn định, hợp dữ liệu lớn & realtime feed.
3. Versioning

Không bao giờ phá vỡ client cũ. URL versioning (/v1/users) — rõ ràng, phổ biến. Header versioning (Accept: application/vnd.api.v2+json) — sạch URL hơn. Quy tắc: thêm field là tương thích; đổi/xoá field là breaking → phải lên version.

4. Idempotency & error format nhất quán
  • Thao tác tạo/thanh toán nhận Idempotency-Key (xem bài Idempotency).
  • Lỗi trả cùng một cấu trúc ở mọi endpoint: { "error": { "code": "...", "message": "...", "details": [...] } } — để client xử lý đồng nhất.
  • Rate limit trả header X-RateLimit-Remaining, Retry-After.
Tư duy Architect: API tốt = nhất quán, dễ đoán, ổn định, có version. Stripe nổi tiếng vì điều này (xem case #6). Contract rõ ràng giảm chi phí tích hợp & bảo trì cho cả hệ sinh thái phụ thuộc bạn.
Time, Clocks & Ordering · "lúc nào" trong hệ phân tán rất khó▶
Vấn đề: không thể tin đồng hồ tường (wall clock)

Mỗi máy có đồng hồ riêng, lệch nhau (clock skew) dù có NTP. Nên không thể dùng timestamp để xác định thứ tự sự kiện giữa các máy: sự kiện X trên máy A "trước" Y trên máy B theo timestamp, nhưng thực tế có thể ngược lại. Điều này gây bug "last write wins" sai, log lệch thứ tự, dữ liệu ghi đè nhầm.

Các loại "thời gian"
  • Physical / wall clock: thời gian thực (UTC). Dùng hiển thị, nhưng không đáng tin để sắp thứ tự nhân quả.
  • Logical clock (Lamport): một bộ đếm tăng dần theo sự kiện & message. Cho biết "A xảy ra trước B" (quan hệ nhân quả) mà không cần đồng hồ đồng bộ. Nhưng không cho biết 2 sự kiện độc lập cái nào trước.
  • Vector clock: mỗi node giữ một vector đếm → phát hiện được sự kiện nào nhân quả, sự kiện nào đồng thời (concurrent → conflict). Dùng trong Dynamo/Cassandra để phát hiện xung đột ghi.
  • Hybrid Logical Clock (HLC) / TrueTime: kết hợp physical + logical. Google Spanner dùng TrueTime (đồng hồ nguyên tử + GPS, cho biết khoảng sai số) để có thứ tự toàn cục ở quy mô toàn cầu.
Đồng hồ lệch khiến timestamp đánh lừa thứ tự thật. Vì vậy hệ phân tán dùng logical/vector clock cho nhân quả, không tin wall clock để quyết định "ai mới hơn".
Ý nghĩa thực tế: Đừng dùng updated_at timestamp để giải xung đột ghi giữa các node. Dùng version/vector clock, hoặc tập trung hoá việc cấp thứ tự (1 leader, hoặc sequencer như Snowflake ID có timestamp + node id). Hiểu điều này giúp tránh bug "mất dữ liệu ghi đè" tinh vi trong hệ phân tán.

4 Architecture Styles & Design

"Đặc sản" của Architect: chọn kiểu kiến trúc, vẽ ra, viết lại quyết định cho người khác hiểu.

Monolith vs Microservices vs Modular Monolith · quyết định lớn nhất▶
Ba lựa chọn
  • Monolith: toàn bộ ứng dụng trong 1 codebase/deploy. Đơn giản phát triển, dễ test, transaction dễ. Nhược khi lớn: build chậm, deploy rủi ro, khó scale từng phần, team dẫm chân nhau.
  • Microservices: chia thành nhiều service nhỏ, deploy độc lập, mỗi service một DB. Lợi: scale & deploy độc lập, team tự chủ, fault isolation. Hại: độ phức tạp vận hành khổng lồ — network, distributed transaction, observability, eventual consistency.
  • Modular Monolith: một deploy nhưng chia module có ranh giới rõ trong code. Lấy phần lớn lợi ích tổ chức code của microservices mà không chịu chi phí phân tán.
Lời khuyên Architect (rất quan trọng): Bắt đầu bằng monolith/modular monolith, tách microservice chỉ khi có lý do thật (team quá lớn, một phần cần scale riêng biệt, vòng đời deploy khác nhau). "Microservices first" cho dự án nhỏ là sai lầm kinh điển. Quy mô tổ chức (Conway's Law) quyết định kiến trúc nhiều hơn ta tưởng.
Monolith: 1 codebase/deploy, module trong cùng tiến trình, 1 DB — đơn giản. Microservices: tách thành nhiều service deploy/scale độc lập, mỗi cái DB riêng — linh hoạt nhưng phức tạp vận hành lớn. Bắt đầu từ monolith.
Domain-Driven Design (DDD) · chia hệ thống theo nghiệp vụ▶
Ý tưởng cốt lõi

Thiết kế phần mềm bám sát domain (nghiệp vụ) thực tế, không theo tầng kỹ thuật. Giúp trả lời câu hỏi sống còn: "chia service/module theo cái gì?".

Khái niệm cần nắm
  • Bounded Context: ranh giới rõ ràng nơi một mô hình/ngôn ngữ có nghĩa nhất quán. Ví dụ "Order" trong context Bán hàng khác "Order" trong context Kho. Mỗi bounded context là ứng viên tự nhiên cho một microservice.
  • Ubiquitous Language: dev và business dùng chung một bộ từ vựng, phản ánh trong code. Giảm hiểu lầm.
  • Entity / Value Object / Aggregate: Aggregate là cụm object thay đổi cùng nhau, có một "aggregate root" làm cổng vào — đảm bảo tính nhất quán cục bộ.
Vì sao Architect cần: Ranh giới service vẽ sai → microservices "phân tán nhưng vẫn dính chặt" (distributed monolith) — tệ nhất mọi thế giới. DDD cho bạn nguyên tắc chia đúng: theo bounded context, không theo bảng DB hay tầng kỹ thuật.
Cùng từ "Order" mang nghĩa khác nhau trong mỗi Bounded Context. Mỗi context có mô hình & ngôn ngữ riêng nhất quán → là ranh giới tự nhiên để tách service (tránh distributed monolith).
Layered / Hexagonal / Clean Architecture · tổ chức code trong 1 service▶
Vấn đề chung các kiến trúc này giải

Làm sao để logic nghiệp vụ độc lập với chi tiết kỹ thuật (DB, framework, API) — để đổi DB hay framework không phải viết lại nghiệp vụ, và test nghiệp vụ không cần DB thật.

  • Layered: chia tầng Presentation → Business → Data. Đơn giản, phổ biến, nhưng dễ rò rỉ chi tiết DB lên trên.
  • Hexagonal (Ports & Adapters): core nghiệp vụ ở giữa, giao tiếp ra ngoài qua port (interface); DB/API/UI là adapter cắm vào. Đổi adapter không đụng core.
  • Clean Architecture: các vòng tròn đồng tâm, dependency luôn trỏ vào trong (tầng ngoài phụ thuộc tầng trong, không ngược lại). Domain ở lõi không biết gì về DB/web.
Nguyên tắc chung: Dependency Inversion — nghiệp vụ định nghĩa interface, hạ tầng cài đặt nó. Đây là chìa khoá làm code testable và bền vững trước thay đổi công nghệ. Với 5 năm kinh nghiệm, đây là thứ nâng bạn từ "viết code chạy được" lên "thiết kế code sống lâu".
Core Domain ở giữa, độc lập kỹ thuật. Mọi thứ bên ngoài (UI, DB, API) là adapter cắm vào qua port (interface). Đổi hạ tầng không phá nghiệp vụ → testable, bền trước thay đổi công nghệ.
C4 Model — vẽ kiến trúc đúng cách · kỹ năng giao tiếp số 1▶
Là gì

C4 là cách vẽ sơ đồ kiến trúc theo 4 mức độ zoom, để mỗi đối tượng người xem thấy đúng mức chi tiết họ cần:

  • 1. Context: hệ thống của bạn + người dùng + hệ thống ngoài. Cho business/non-tech. "Bức tranh lớn".
  • 2. Container: các ứng dụng/datastore lớn (web app, mobile, API, DB, queue) và cách chúng nói chuyện. Cho kỹ sư.
  • 3. Component: bên trong một container có những module/component nào.
  • 4. Code: class/diagram chi tiết (ít khi cần vẽ tay, hiếm dùng).
Vì sao Architect bắt buộc thạo: Một thiết kế chỉ có giá trị khi người khác hiểu và đồng thuận. C4 chuẩn hoá cách trình bày, tránh sơ đồ "mỗi người vẽ một kiểu". Công cụ: Excalidraw, draw.io, Mermaid, Structurizr. Luyện vẽ C4 cho chính hệ thống bạn đang làm.
C4 = vẽ kiến trúc theo 4 mức zoom: Context (bức tranh lớn) → Container → Component → Code. Mỗi mức cho đúng đối tượng cần xem → chuẩn hoá giao tiếp kiến trúc.
ADR & RFC — ghi lại quyết định · tài liệu của Architect▶
ADR (Architecture Decision Record)

Một tài liệu ngắn ghi lại một quyết định kiến trúc quan trọng, theo cấu trúc:

  • Context: bối cảnh, vấn đề cần giải, ràng buộc.
  • Decision: đã quyết chọn gì.
  • Alternatives: các phương án khác đã cân nhắc.
  • Consequences: hệ quả tốt/xấu, đánh đổi chấp nhận.
RFC (Request for Comments) / Design Doc

Tài liệu dài hơn, đề xuất một thiết kế/thay đổi lớn để cả team góp ý trước khi làm. Thường có: mục tiêu & non-goals, các phương án + trade-off, sơ đồ, rủi ro, kế hoạch triển khai.

Vì sao quan trọng nhất: Quyết định kiến trúc sẽ bị quên "tại sao" sau 6 tháng. ADR giữ lại lý do để người sau (và chính bạn) không "đập đi xây lại" mù quáng. Khả năng viết ADR/RFC rõ ràng là thứ phân biệt rõ nhất giữa Senior Engineer và Architect.
API Gateway, BFF & Service Mesh · hạ tầng giao tiếp của microservices▶
Vấn đề

Khi có hàng chục service, client không thể tự biết gọi service nào, và mỗi service không nên tự lo auth/rate-limit/logging riêng lẻ. Ba khái niệm dưới đây giải quyết "giao tiếp ở giữa".

  • API Gateway: một cổng vào duy nhất cho client. Lo các việc "cross-cutting": xác thực, rate limiting, định tuyến tới service đúng, tổng hợp response, SSL. Client chỉ nói chuyện với gateway. (North-south traffic — client ↔ hệ thống.)
  • BFF (Backend for Frontend): mỗi loại client (web, mobile, TV) có một gateway riêng tối ưu cho nhu cầu của nó — tránh một API "ôm đồm" làm khổ mọi client.
  • Service Mesh (Istio, Linkerd): hạ tầng lo giao tiếp giữa các service với nhau (east-west traffic) — mTLS, retry, circuit breaker, tracing — được "tiêm" qua sidecar proxy nên code service không phải tự cài. Giảm trùng lặp logic resilience khắp nơi.
Đừng over-engineer: Service Mesh rất mạnh nhưng thêm độ phức tạp vận hành lớn (mỗi pod có sidecar). Chỉ cần khi đã có nhiều service. Vài service thì một API Gateway + thư viện resilience là đủ.
Phân biệt nhanh: API Gateway = cửa ngõ client vào (north-south). Service Mesh = giao thông nội bộ giữa các service (east-west). BFF = gateway chuyên biệt cho từng loại frontend.
API Gateway = cổng vào cho client (north-south): auth, rate limit, routing. Service Mesh = lo giao thông giữa các service (east-west): mTLS, retry, circuit breaker qua sidecar, không cần sửa code service.
Multi-tenancy — phục vụ nhiều khách trên một hệ · cốt lõi của SaaS▶
Là gì & vì sao quan trọng

SaaS phục vụ nhiều khách hàng (tenant) trên cùng hạ tầng. Câu hỏi kiến trúc lớn: cô lập dữ liệu giữa các tenant tới mức nào? Đây là quyết định khó sửa, ảnh hưởng bảo mật, chi phí, vận hành.

3 mô hình: Silo (DB riêng mỗi tenant — cô lập mạnh, đắt), Bridge (schema riêng — cân bằng), Pool (chung bảng + tenant_id — rẻ, scale tốt nhưng phải tuyệt đối không quên filter theo tenant_id).
Quyết định & cạm bẫy
  • Pool model (shared, tenant_id) phổ biến nhất vì rẻ & scale — nhưng rủi ro lớn nhất: quên filter WHERE tenant_id=? → rò rỉ dữ liệu khách này sang khách khác (sự cố bảo mật nghiêm trọng). Giải: ép ở tầng ORM/middleware, dùng Row-Level Security của Postgres.
  • Silo cho khách enterprise yêu cầu cô lập/tuân thủ cao; Pool cho khách nhỏ số lượng lớn. Nhiều SaaS dùng lai: pool cho đa số, silo cho khách lớn.
  • Noisy neighbor: một tenant ngốn tài nguyên ảnh hưởng người khác → cần rate limit/quota per-tenant.

5 Reliability, Security & Observability

"Everything fails all the time." Architect được đánh giá qua cách hệ thống chịu lỗi và được vận hành.

"Khi 3 giờ sáng một service downstream chết, hệ thống của tôi sụp đổ hay tự co lại duyên dáng?"

Resilience patterns: Timeout, Retry, Circuit Breaker, Bulkhead · chống sập dây chuyền▶
Vấn đề: Cascading Failure

Trong hệ phân tán, một service chậm có thể kéo sập cả hệ thống: các caller chờ nó → cạn thread/connection → caller cũng chậm → lan ra toàn bộ. Các pattern dưới đây phải dùng kết hợp để chặn dây chuyền này.

  • Timeout: không bao giờ chờ vô hạn. Đặt thời hạn cho mọi cuộc gọi ra ngoài. Thiếu timeout là nguyên nhân cascading failure số 1.
  • Retry + Backoff + Jitter: thử lại khi lỗi tạm thời, nhưng tăng dần khoảng chờ (exponential backoff) và thêm ngẫu nhiên (jitter) để tránh hàng ngàn client retry cùng lúc (retry storm). Chỉ retry thao tác idempotent!
  • Circuit Breaker: như cầu dao điện. Nếu service B lỗi liên tục, "ngắt mạch" — ngừng gọi B một thời gian, trả lỗi/fallback ngay, cho B thời gian hồi phục. Ba trạng thái: Closed (bình thường) → Open (ngắt) → Half-Open (thử lại dè dặt).
  • Bulkhead: như khoang kín tàu thuỷ — cô lập tài nguyên (thread pool riêng cho từng downstream) để một phần chết không làm chìm cả tàu.
Vòng đời Circuit Breaker: Closed → (lỗi nhiều) → Open → (chờ) → Half-Open → (OK) → Closed.
Chốt: Timeout + Retry + Circuit Breaker là bộ ba bắt buộc cho mọi cuộc gọi inter-service. Hiểu vì sao thiếu một cái sẽ gây thảm hoạ là dấu hiệu tư duy reliability trưởng thành.
Rate Limiting, Throttling & Backpressure · kiểm soát tải vào▶
Rate Limiting

Giới hạn số request mỗi client trong một khoảng thời gian — để chống lạm dụng, DDoS, và bảo vệ hệ thống khỏi quá tải. Các thuật toán:

  • Token Bucket: mỗi client có "xô" token nạp đều theo thời gian; mỗi request tiêu 1 token; hết token → từ chối. Cho phép burst. Phổ biến nhất.
  • Leaky Bucket: xử lý request với tốc độ cố định, làm mượt traffic.
  • Fixed / Sliding Window: đếm request theo cửa sổ thời gian.
Backpressure

Khi consumer xử lý không kịp, nó "đẩy ngược" tín hiệu cho producer chậm lại, thay vì âm thầm phình bộ nhớ rồi sập. Quan trọng trong stream processing và hệ async.

Token Bucket: xô được nạp token đều theo thời gian; mỗi request tiêu 1 token. Còn token → cho qua (cho phép burst tới sức chứa xô); hết → trả 429. Đơn giản, ít bộ nhớ, phổ biến nhất.
Chốt: Hệ thống tốt từ chối có kiểm soát khi quá tải (trả 429) thay vì sập hoàn toàn. Bảo vệ phần lõi quan trọng hơn phục vụ mọi request.
Observability: Logs, Metrics, Traces & SLO · nhìn thấy bên trong hệ thống▶
Ba trụ cột (three pillars)
  • Logs: bản ghi sự kiện rời rạc. Nên structured (JSON) để query được. Trả lời "chuyện gì đã xảy ra lúc đó?".
  • Metrics: số liệu tổng hợp theo thời gian (QPS, latency, error rate, CPU). Rẻ, hợp cho dashboard & alert. Khung phổ biến: RED (Rate, Errors, Duration) cho service, USE (Utilization, Saturation, Errors) cho tài nguyên.
  • Traces (distributed tracing): theo dấu một request đi qua nhiều service, mỗi chặng là một "span" gắn cùng trace ID. Trả lời "request chậm ở khâu nào?". Cốt lõi để debug microservices.
SLO & Error Budget

SLO = mục tiêu (vd 99.9% request < 300ms). Error budget = phần được phép lỗi (0.1%). Nếu còn budget → cứ ship feature; nếu cháy budget → dừng feature, dồn sửa reliability. Đây là cách dùng số liệu để cân bằng tốc độ và ổn định một cách khách quan.

3 trụ cột: Logs (sự kiện), Metrics (số liệu tổng hợp — RED/USE), Traces (theo dấu 1 request qua nhiều service bằng trace ID). Cùng nuôi dashboard & alert để vừa giám sát vừa điều tra.
Phân biệt: Monitoring = theo dõi cái bạn biết trước cần xem. Observability = khả năng điều tra cái chưa biết từ dữ liệu hệ thống phát ra. Hệ phân tán bắt buộc phải observable mới vận hành nổi.
AuthN vs AuthZ, OAuth2/OIDC, JWT · danh tính & quyền▶
Hai khái niệm hay nhầm
  • Authentication (AuthN): "bạn là ai?" — xác minh danh tính (đăng nhập).
  • Authorization (AuthZ): "bạn được làm gì?" — kiểm tra quyền (RBAC theo vai trò, ABAC theo thuộc tính).
OAuth2 & OIDC

OAuth2 là framework uỷ quyền (cho app A truy cập tài nguyên của bạn ở B mà không đưa mật khẩu). Flow hay gặp: Authorization Code (cho web/mobile có user), Client Credentials (service-to-service, không có user). OIDC xây trên OAuth2 để thêm phần xác thực danh tính (trả về ID token cho biết user là ai).

JWT

Token tự chứa (self-contained), gồm header.payload.signature, được ký để chống giả mạo. Server verify chữ ký mà không cần tra DB → hợp cho stateless auth. Nhược: khó thu hồi trước hạn (đã phát là có hiệu lực tới khi hết hạn) → giữ thời hạn ngắn + dùng refresh token; không nhét dữ liệu nhạy cảm vào payload (chỉ được encode, không mã hoá).

OAuth2 Authorization Code: user đăng nhập ở Auth Server, app nhận "code" rồi đổi lấy access token, dùng token gọi Resource API. App không chạm mật khẩu. OIDC thêm ID token cho biết "user là ai".
Security cốt lõi: Threat Modeling, Encryption, OWASP · nghĩ như kẻ tấn công▶
Threat Modeling (STRIDE)

Quy trình chủ động tìm điểm yếu trước khi bị khai thác. Khung STRIDE liệt kê 6 loại đe doạ: Spoofing (giả danh), Tampering (sửa dữ liệu), Repudiation (chối bỏ), Information disclosure (lộ tin), Denial of service, Elevation of privilege (leo thang quyền). Với mỗi luồng dữ liệu, hỏi "kẻ xấu làm được gì ở đây?".

Encryption
  • In transit: mã hoá đường truyền — TLS/HTTPS bắt buộc.
  • At rest: mã hoá dữ liệu lưu trữ (DB, file, backup).
  • Secrets management: không hardcode API key/mật khẩu trong code — dùng Vault, KMS, secret manager.
OWASP Top 10 & Zero Trust

Thuộc OWASP Top 10 (injection, broken auth, lộ dữ liệu nhạy cảm, SSRF...) — danh sách lỗ hổng web phổ biến nhất. Zero Trust: "không tin mặc định" — xác thực & phân quyền mọi request kể cả trong mạng nội bộ, không coi "ở trong mạng" là an toàn.

Tư duy Architect: Bảo mật phải thiết kế từ đầu (security by design), không vá sau. Mỗi thành phần nhận input từ ngoài là một bề mặt tấn công cần được mô hình hoá.
Cloud, Containers, Kubernetes & IaC · nền hạ tầng hiện đại▶
Containers

Container (Docker) đóng gói app + mọi dependency thành một đơn vị chạy nhất quán "máy nào cũng giống nhau". Nhẹ hơn VM (chia sẻ kernel OS). Giải quyết "máy tôi chạy được mà server không".

Kubernetes (K8s)

Hệ điều phối container ở quy mô lớn: tự động deploy, scale, self-heal (container chết tự khởi động lại), rolling update, service discovery, load balancing nội bộ. Khái niệm: Pod (đơn vị chạy), Deployment (quản lý số bản sao), Service (điểm truy cập ổn định).

IaC & Multi-region
  • Infrastructure as Code (Terraform): mô tả hạ tầng bằng code, version control, tái lập được — không "click chuột thủ công".
  • Multi-AZ / Multi-region: đặt hệ thống ở nhiều vùng để chịu được sự cố cả một datacenter; phục vụ user toàn cầu với latency thấp.
  • FinOps / Cost awareness: Architect phải cân nhắc chi phí — kiến trúc "đúng" mà đốt tiền vô lý là kiến trúc sai.
K8s điều phối container: Pod (đơn vị chạy), Deployment (giữ N bản sao & rolling update), Service (điểm vào ổn định + LB nội bộ). Pod chết → tự khởi động lại (self-heal). IaC mô tả tất cả bằng code.
Cost & Capacity Modeling (FinOps) · kiến trúc đốt tiền là kiến trúc sai▶
Vì sao Architect phải nghĩ về chi phí

Một thiết kế "đúng kỹ thuật" nhưng tốn gấp 10 lần cần thiết là thiết kế tồi. Chi phí là một NFR thật sự. Architect giỏi cân hiệu năng/độ bền với hoá đơn cloud.

Capacity planning — ước lượng tài nguyên cần
  • Từ estimation (QPS, storage — GĐ1) → suy ra số server, IOPS, băng thông, RAM cache cần.
  • Chừa headroom cho spike (đừng chạy 100% công suất) nhưng đừng over-provision lãng phí.
  • Tính cho peak, không phải trung bình — nhưng dùng auto-scaling để không trả tiền cho peak suốt ngày.

6 Vai trò Architect & thực chiến

Kiến thức kỹ thuật chỉ là một nửa. Nửa còn lại là tư duy tổng thể, ra quyết định và sức ảnh hưởng.

"Làm sao thuyết phục cả team và business đi theo một quyết định kỹ thuật — và chịu trách nhiệm cho nó?"

Trade-off Analysis có hệ thống · kỹ năng lõi của Architect▶
Là gì

Không có giải pháp hoàn hảo — mọi quyết định kiến trúc đều đánh đổi. Architect giỏi là người nêu rõ đánh đổi và chọn có lý do, không phải người "biết câu trả lời đúng".

Khung phân tích
  • Liệt kê 2–3 phương án khả thi (luôn có ít nhất một phương án "không làm gì / đơn giản nhất").
  • Định nghĩa tiêu chí đánh giá theo NFR (xem bài sau): hiệu năng, chi phí, độ phức tạp vận hành, time-to-market, khả năng mở rộng, rủi ro.
  • Lập bảng cho điểm từng phương án theo tiêu chí. Nêu rõ cái gì được, cái gì mất.
  • Chọn + ghi vào ADR. Gắn với ràng buộc hiện tại, không phải lý tưởng.
Câu thần chú: "It depends — và đây là nó phụ thuộc vào cái gì." Trả lời được vế sau là dấu hiệu trưởng thành. Tránh cả hai cực: cứng nhắc giáo điều và "tuỳ" mơ hồ.
Non-Functional Requirements (NFR) · yêu cầu phi chức năng▶
Là gì

Functional requirement = "hệ thống làm gì" (đặt hàng, gửi tin). NFR = "hệ thống tốt thế nào" — và đây mới là thứ định hình kiến trúc. Gồm: scalability, availability, latency/performance, reliability, security, maintainability, cost, compliance.

Vì sao quan trọng nhất với Architect

Hai hệ thống cùng chức năng (ví dụ "chat") nhưng NFR khác nhau (1000 user vs 1 tỉ user, latency 1s vs 50ms) sẽ có kiến trúc hoàn toàn khác. Khi nhận yêu cầu, Architect phải khai thác NFR trước bằng câu hỏi: bao nhiêu user? đọc nhiều hay ghi nhiều? chấp nhận trễ bao nhiêu? consistency mức nào? ngân sách?

Chốt: Bắt đầu mọi bài thiết kế bằng việc làm rõ NFR. Bỏ qua bước này là sai lầm lớn nhất trong cả phỏng vấn lẫn thực tế — bạn sẽ thiết kế sai quy mô.
Migration & Evolution — Strangler Fig · thay đổi hệ thống đang chạy▶
Vấn đề

Phần lớn công việc Architect không phải xây mới mà tiến hoá hệ thống cũ đang phục vụ khách hàng — không được dừng, không được mất dữ liệu. "Big bang rewrite" (viết lại toàn bộ rồi đổi một phát) hầu như luôn thất bại.

Strangler Fig Pattern

Lấy ý tưởng từ cây bóp cổ: bọc hệ cũ, dần dần thay từng phần bằng hệ mới, định tuyến từng tính năng sang hệ mới, cho đến khi hệ cũ "teo" hoàn toàn rồi gỡ bỏ. Lợi: rủi ro thấp, làm dần, rollback dễ ở từng bước.

Kèm theo cần
  • Feature flags / canary release: bật tính năng cho một phần nhỏ user trước.
  • Kế hoạch rollback rõ ràng cho mỗi bước.
  • Dual-write / backfill khi migrate dữ liệu giữa hệ cũ và mới.
Stakeholder Communication & "ngôn ngữ business" · ảnh hưởng & thuyết phục▶
Vì sao đây là kỹ năng quyết định thăng tiến

Một thiết kế xuất sắc mà không thuyết phục được team/CTO/PO thì vô dụng. Càng lên cao, công việc càng ít code, càng nhiều giao tiếp và ra ảnh hưởng. Đây thường là rào cản lớn nhất khi Senior lên Architect.

Cách làm
  • Dịch kỹ thuật sang business: đừng nói "ta cần sharding", hãy nói "với cách này, khi user tăng gấp 10 chi phí chỉ tăng 2 lần và không cần downtime". Nói bằng chi phí, rủi ro, doanh thu, time-to-market.
  • Biết khán giả: với CTO nói chiến lược & rủi ro; với dev nói chi tiết kỹ thuật; với PO nói tác động tới tính năng & lịch.
  • Trình bày phương án + khuyến nghị, không chỉ vấn đề. Lãnh đạo muốn lựa chọn rõ ràng kèm đánh đổi.
The Architect Elevator (Gregor Hohpe): Architect phải "đi thang máy" lên xuống giữa tầng lãnh đạo (chiến lược) và tầng kỹ thuật (engine room), kết nối hai thế giới. Người ở mãi một tầng không thể là Architect tốt.

Dẫn dắt Design Review · kỹ năng đòn bẩy của Architect▶
Vì sao quan trọng

Architect tạo ảnh hưởng không phải bằng cách tự thiết kế mọi thứ, mà bằng cách nâng chất lượng thiết kế của cả tổ chức qua review. Một design review tốt phát hiện vấn đề khi sửa còn rẻ (trên giấy), không phải sau khi đã build.

Cách chạy một review hiệu quả
  • Đọc trước, họp sau: tài liệu (RFC) gửi trước, mọi người đọc & comment; buổi họp chỉ bàn điểm tranh luận — không đọc slide tại chỗ.
  • Hỏi câu mở, không phán xét: "Điều gì xảy ra khi service X chết?", "Tại sao chọn cách này thay vì Y?", "Phần nào em lo nhất?" — giúp tác giả tự thấy lỗ hổng thay vì bị áp đặt.
  • Tập trung vào điều khó sửa sau: ranh giới dữ liệu, API contract, mô hình consistency — không sa đà vào tiểu tiết code.
  • Phân biệt "chặn" vs "góp ý": nêu rõ đâu là vấn đề phải sửa (blocking) đâu chỉ là gợi ý (nit), để tác giả không tê liệt.
  • An toàn tâm lý: review thiết kế, không review con người. Mục tiêu là thiết kế tốt hơn, không phải chứng tỏ ai giỏi hơn.
Checklist câu hỏi vàng khi review: NFR rõ chưa? · Failure mode & resilience? · Dữ liệu nhất quán thế nào? · Có over-engineering không? · Phương án đơn giản hơn? · Migration/rollout ra sao? · Đo lường/observability? · Bảo mật?
Tech Strategy, Build-vs-Buy & Tech Radar · định hướng dài hạn▶
Build vs Buy — quyết định Architect hay phải làm

Tự xây hay dùng giải pháp có sẵn (SaaS/open-source)? Khung quyết định:

  • Có phải lợi thế cạnh tranh cốt lõi không? Nếu CÓ → cân nhắc build (kiểm soát, khác biệt). Nếu KHÔNG (auth, email, payment, search hạ tầng) → thường buy/dùng có sẵn, đừng phát minh lại bánh xe.
  • Tổng chi phí sở hữu (TCO): build không chỉ là code ban đầu mà còn vận hành, bảo trì, on-call nhiều năm. "Free" thường không free.
  • Time-to-market: buy nhanh hơn để ra thị trường; build chậm nhưng kiểm soát.
  • Rủi ro khoá nhà cung cấp (vendor lock-in) vs rủi ro tự vận hành.
Tech Radar & chiến lược công nghệ

Architect duy trì một "radar" công nghệ: phân loại Adopt (dùng tự tin), Trial (thử có kiểm soát), Assess (theo dõi), Hold (tránh/loại bỏ). Giúp tổ chức nhất quán, tránh mỗi team một stack hỗn loạn. Gắn lựa chọn công nghệ với mục tiêu business & năng lực đội ngũ, không theo trend.

Tư duy chiến lược: Quyết định công nghệ tốt cân cả 3: nhu cầu business · năng lực & quy mô đội · chi phí dài hạn. Nhớ "Choose Boring Technology" — tiêu "token đổi mới" vào đúng chỗ tạo giá trị.
Influence without authority & Mentoring · nhân sức ảnh hưởng▶
Ảnh hưởng không cần quyền lực

Architect thường không quản lý ai nhưng phải khiến nhiều team đi cùng hướng. Cách tạo ảnh hưởng:

  • Uy tín kỹ thuật + lắng nghe: hiểu sâu vấn đề của từng team trước khi đề xuất; người ta theo người vừa giỏi vừa hiểu họ.
  • Thuyết phục bằng dữ liệu & trade-off, không bằng chức danh. Cho người khác tham gia vào quyết định → họ sở hữu nó.
  • Chọn trận để đánh: không phải mọi điểm đều đáng tranh luận. Giữ vốn ảnh hưởng cho quyết định thực sự quan trọng & khó đảo ngược.
  • Viết ra (RFC, ADR): ý tưởng viết rõ lan xa hơn lời nói & tồn tại lâu hơn.
Mentoring — đòn bẩy lớn nhất

Cách nhân năng lực không phải tự làm nhiều hơn mà là nâng người khác lên. Dạy tư duy (cách hỏi, cách phân tích trade-off) thay vì cho đáp án. Một Architect giỏi để lại sau lưng cả một thế hệ kỹ sư biết tư duy thiết kế — đó là di sản & cũng là thước đo thật sự.

Chuyển dịch tư duy khi lên Architect: từ "tôi giải quyết vấn đề" → "tôi giúp tổ chức giải quyết vấn đề tốt hơn & bền hơn". Thành công đo bằng tác động qua người khác, không phải số dòng code bạn viết.

🧓 Bài học xương máu từ tiền bối

Những nguyên tắc đã trả giá bằng sự cố production của hàng trăm hệ thống. Đây là phần "trực giác" mà sách giáo khoa ít dạy — nhưng phân biệt rõ nhất một Architect dày dạn với người mới.

Chọn công nghệ "nhàm chán" (Choose Boring Technology) Dan McKinley▶
"Mỗi công ty có một số ít 'token đổi mới'. Đừng tiêu hết chúng vào tầng hạ tầng. Hãy dùng công nghệ nhàm chán, đã được kiểm chứng, cho phần lớn hệ thống."— Dan McKinley (cựu kỹ sư Etsy)
Ý nghĩa

Công nghệ mới luôn hấp dẫn, nhưng đi kèm "ẩn số chưa biết" (failure mode lạ, ít người biết debug, cộng đồng nhỏ). Postgres, MySQL, Redis "nhàm chán" vì chúng đã được hàng triệu người dùng đào hết hố — bạn biết trước nó hỏng thế nào.

Áp dụng: Mỗi lựa chọn công nghệ mới là một "khoản nợ vận hành". Chỉ tiêu "token đổi mới" vào thứ tạo lợi thế cạnh tranh thật cho business, phần còn lại chọn thứ buồn tẻ và đáng tin. Tránh "resume-driven development" (chọn tech vì đẹp CV).
Mọi thứ đều sẽ hỏng — thiết kế cho thất bại Werner Vogels · Amazon CTO▶
"Everything fails, all the time."— Werner Vogels (CTO Amazon)
Ý nghĩa

Ở quy mô lớn, phần cứng hỏng, mạng đứt, dependency timeout là chuyện thường ngày, không phải ngoại lệ. Câu hỏi không phải "nếu hỏng" mà "khi hỏng thì sao". Hệ thống tốt không phải hệ không bao giờ lỗi, mà là hệ chịu lỗi mà không sập dây chuyền.

Cùng triết lý này Amazon/Netflix sinh ra Chaos Engineering — chủ động gây lỗi (tắt máy ngẫu nhiên) trong production để ép hệ thống phải bền ngay từ thiết kế.

"You build it, you run it."— Werner Vogels (về DevOps & ownership)

Đội xây service cũng vận hành nó (on-call). Khi tự chịu hậu quả của thiết kế tồi, người ta thiết kế tốt hơn.

Monolith trước, microservices sau Martin Fowler▶
"Đừng bắt đầu một dự án mới bằng microservices, kể cả khi bạn chắc ứng dụng sẽ đủ lớn để cần. Bạn phải 'đủ cao' mới được chơi microservices."— Martin Fowler (MonolithFirst / "You must be this tall…")
Ý nghĩa

Lúc đầu bạn chưa biết ranh giới service đúng nằm đâu. Tách quá sớm = vẽ sai ranh giới = "distributed monolith" (các service vẫn dính chặt nhưng giờ phải gọi qua mạng) — tệ nhất mọi thế giới. Monolith cho bạn refactor ranh giới rẻ và nhanh tới khi hiểu rõ domain.

Điều kiện "đủ cao" để dùng microservices: có CI/CD tốt, monitoring/observability, khả năng provision nhanh, và tổ chức đủ lớn để mỗi team sở hữu một service. Thiếu những thứ này, microservices sẽ giết tốc độ của startup.
Đơn giản & chống tối ưu hoá sớm Knuth · Kelsey Hightower▶
"Premature optimization is the root of all evil."— Donald Knuth
Ý nghĩa

Đừng thêm cache/shard/queue khi chưa đo được rằng có nút thắt thật. Mỗi thành phần thêm vào là độ phức tạp vĩnh viễn. "Make it work, make it right, make it fast" — theo đúng thứ tự đó.

"Thiết kế cho tải gấp 10 lần, nhưng dự liệu phải thiết kế lại khi tải gấp 100 lần."— Triết lý scaling phổ biến (Jeff Dean / Google school)

Đừng thiết kế cho "1 tỉ user" khi bạn có 1000. Thiết kế cho mức tăng hợp lý kế tiếp, và quan trọng hơn: đừng đưa ra quyết định chặn đường việc scale sau này (ví dụ: chọn shard key cho phép tách sau).

Dữ liệu sống lâu hơn code Kinh nghiệm chung▶
Ý nghĩa

Code được viết lại liên tục, nhưng dữ liệu và schema thì ở lại nhiều năm. Quyết định khó sửa nhất, đắt nhất nếu sai, luôn là: mô hình dữ liệu, shard key, ranh giới ownership dữ liệu giữa các service.

Áp dụng: Dành nhiều thời gian nhất cho thiết kế dữ liệu. Một API xấu sửa trong một sprint; một shard key sai có thể buộc migration đau đớn hàng tháng. "Show me your data structures and I won't usually need your code" (Fred Brooks / Linus Torvalds).
Conway's Law — kiến trúc phản chiếu tổ chức Melvin Conway · Sam Newman▶
"Một tổ chức thiết kế hệ thống sẽ tạo ra thiết kế sao chép đúng cấu trúc giao tiếp của chính tổ chức đó."— Melvin Conway
Ý nghĩa

Nếu bạn có 4 team, bạn gần như chắc chắn sẽ kết thúc với kiến trúc 4 phần — dù muốn hay không. Ranh giới service có xu hướng bám theo ranh giới team.

Inverse Conway Maneuver: Muốn có kiến trúc thế nào, hãy tổ chức team theo hình dạng đó trước. Đây là lý do "two-pizza team" (Amazon) và Team Topologies tồn tại. Architect không chỉ thiết kế hệ thống — phải để ý cả cách team được chia.
Observability trước khi scale · ghi lại "tại sao" Charity Majors▶
Ý nghĩa

Bạn không thể vận hành — càng không thể scale — thứ bạn không nhìn thấy. Tiền bối học được rằng logging/metrics/tracing phải có trước khi hệ thống đủ lớn để gặp sự cố khó, không phải vá sau khi cháy.

"Nguy hiểm không phải cái bạn biết là mình không biết, mà cái bạn tưởng mình biết nhưng thật ra sai."— tinh thần observability (Charity Majors)

Và bài học quản trị: luôn ghi lại "tại sao" (ADR). Sau 6 tháng không ai nhớ vì sao chọn DB này, dẫn tới "đập đi xây lại" mù quáng. Quyết định không được ghi lại = quyết định sẽ bị lặp lại sai lầm.

Đúc kết một câu

Tiền bối giỏi không phải người biết nhiều pattern nhất, mà là người kỷ luật chọn giải pháp đơn giản nhất đủ dùng, thiết kế cho lúc hỏng, và ghi lại lý do. Sự trưởng thành kỹ thuật đo bằng số thứ bạn không thêm vào, chứ không phải số thứ bạn thêm.

Vì sao học được nhiều từ Alex Xu

Giá trị lớn nhất của sách/ByteByteGo của Alex Xu không phải đáp án, mà là khung tư duy lặp lại áp cho mọi bài. Khi đọc bất kỳ thiết kế nào (của ông ấy hay engineering blog của công ty), hãy "mổ xẻ" theo các câu hỏi:

  • Yêu cầu & NFR là gì? Quy mô bao nhiêu? Đọc nhiều hay ghi nhiều? Consistency cần mức nào?
  • Estimation: con số nào (QPS, storage) dẫn tới quyết định kiến trúc?
  • Nút thắt ở đâu? Họ giải bằng cách gì (cache? shard? queue? CDN?)
  • Họ đánh đổi cái gì? Chọn AP hay CP? Eventual hay strong? Vì sao chấp nhận được trong bối cảnh này?
  • Nếu là mình, mình sẽ làm khác chỗ nào — và tại sao?
Cách luyện: Đọc một bài → gấp lại → tự vẽ lại thiết kế từ đầu theo trí nhớ → so sánh chỗ khác biệt. Đọc thụ động không tạo kỹ năng; tái tạo chủ động mới tạo kỹ năng. Nguồn vàng: ByteByteGo, engineering blog của Netflix/Uber/Discord/Stripe, và "The System Design Primer" trên GitHub.
Instagram — sức mạnh của sự đơn giản đội ~13 người, hàng chục triệu user▶
Bối cảnh

Instagram scale tới hàng chục triệu user với một đội kỹ sư cực nhỏ, chạy trên PostgreSQL + AWS. Triết lý nổi tiếng: "do the simple thing first", "giữ ít công nghệ nhất có thể".

Vấn đề & cách họ giải
  • Scale ghi vượt một Postgres: họ dùng logical sharding — tạo hàng nghìn "shard logic" rồi ánh xạ vào số ít máy vật lý. Khi cần mở rộng, chỉ việc di chuyển shard logic sang máy mới mà không đổi code ứng dụng.
  • Sinh ID phân tán: tự thiết kế ID 64-bit nhúng cả timestamp + shard id + sequence → ID sắp xếp được theo thời gian mà không cần node trung tâm.
  • Dùng Redis/Memcached cho cache, giữ stack nhỏ và dễ vận hành.
Bài học: Bạn không cần NoSQL hào nhoáng hay microservices để scale lớn. Một Postgres được shard khéo + cache + ID design tốt đưa bạn đi rất xa. Đội nhỏ thắng nhờ kỷ luật giữ mọi thứ đơn giản.
Discord — lưu trữ hàng nghìn tỉ tin nhắn trillions of messages▶
Bối cảnh & vấn đề

Discord cần lưu & truy vấn lượng tin nhắn tăng theo cấp số nhân. Họ trải qua một hành trình kinh điển về chọn datastore: MongoDB → Cassandra → ScyllaDB.

Cách họ giải / vì sao đổi
  • Rời MongoDB: không kham nổi quy mô & pattern truy cập của họ → cần một store ghi-nặng, scale ngang tốt.
  • Sang Cassandra (wide-column): hợp với "đọc tin theo channel + thời gian". Họ thiết kế partition key theo (channel, khoảng thời gian) để truy vấn chỉ chạm một partition. Nhưng gặp đau đầu với GC pause, "hot partition" (channel khổng lồ), và chi phí vận hành.
  • Sang ScyllaDB: tương thích Cassandra nhưng viết bằng C++/Rust-style, không GC pause → latency p99 ổn định hơn, ít máy hơn.
Bài học: (1) Chọn datastore phải khớp access pattern (đọc theo channel+time → wide-column). (2) Thiết kế partition key để tránh hot partition là sống còn. (3) Kiến trúc tiến hoá — không xấu hổ khi đổi store nếu bằng chứng vận hành cho thấy cần. Họ đo đạc rồi mới đổi, không đổi theo cảm tính.
Netflix — kỹ nghệ chịu lỗi (Resilience Engineering) microservices toàn cầu▶
Bối cảnh

Netflix chạy hàng trăm microservices trên AWS phục vụ toàn cầu. Một service lỗi không được làm sập trải nghiệm xem phim.

Cách họ giải
  • Circuit Breaker (Hystrix): tiên phong đóng gói pattern circuit breaker + bulkhead + fallback thành thư viện dùng chung. Service downstream lỗi → trả fallback (ví dụ danh sách phim mặc định) thay vì sập.
  • Chaos Engineering (Chaos Monkey / Simian Army): chủ động tắt instance ngẫu nhiên trong production để ép mọi service phải bền vững — biến "chịu lỗi" thành yêu cầu mặc định chứ không phải tính năng phụ.
  • Regional failover: có thể "sơ tán" toàn bộ traffic khỏi một region AWS đang gặp sự cố.
Bài học: Độ tin cậy không đến từ "cố gắng không hỏng" mà từ chủ động luyện tập cho lúc hỏng. Fallback & degrade duyên dáng > cố giữ tính năng đầy đủ rồi sập toàn bộ. Đây là tư duy bạn nên mang vào dự án Capstone #8.
Twitter/X — Timeline & bài toán fan-out read-heavy cực lớn▶
Vấn đề

Khi bạn mở app, "home timeline" phải hiện tweet mới nhất của tất cả người bạn follow — tức thì. Đọc gấp hàng trăm lần ghi. Tính timeline lúc đọc (truy vấn tweet của hàng nghìn người rồi trộn) thì quá chậm.

Cách họ giải
  • Fan-out on write: khi một user đăng tweet, đẩy ngay tweet đó vào timeline cache (Redis) của từng follower. Lúc đọc chỉ việc lấy list đã dựng sẵn → cực nhanh.
  • Vấn đề celebrity: người có hàng chục triệu follower → fan-out on write tốn kinh khủng (1 tweet = chục triệu lần ghi). Giải: hybrid — celebrity dùng fan-out on read (lúc bạn mở feed mới trộn tweet của họ vào), user thường dùng fan-out on write.
Bài học: Không có một lời giải cho mọi user — phân loại workload và áp chiến lược khác nhau (hybrid) là dấu hiệu thiết kế trưởng thành. Đây chính xác là bài học bạn sẽ tự chạm trong dự án News Feed #7.
Uber — định vị & ghép chuyến (Geospatial) real-time, theo vị trí▶
Vấn đề

"Tìm tài xế gần khách trong vài trăm mét, real-time, trên toàn thành phố" — truy vấn theo không gian địa lý ở quy mô lớn rất khó với index thông thường.

Cách họ giải
  • H3 — lưới ô lục giác: Uber chia bản đồ thành các ô lục giác (hexagon) nhiều cấp độ phân giải (đã open-source). Mỗi vị trí thuộc một ô → "tìm tài xế gần" trở thành "tìm trong ô này và các ô kề" — cực nhanh, phân tán được theo ô.
  • Vì sao lục giác (không phải ô vuông): khoảng cách tới mọi ô láng giềng đều bằng nhau → tính lân cận chính xác hơn.
  • Kết hợp xử lý real-time để cập nhật vị trí tài xế liên tục.
Bài học: Bài toán đặc thù (geo, search, đồ thị) cần cấu trúc dữ liệu chuyên dụng, không cố nhét vào quan hệ thông thường. Một Architect biết "khi nào pattern thông thường không còn hợp" và tìm công cụ đúng (geospatial index, trie cho autocomplete, graph DB cho quan hệ).
Stripe — Idempotency & API design bậc thầy thanh toán, không sai một xu▶
Vấn đề

Thanh toán: mạng chập chờn, client retry → tuyệt đối không được trừ tiền hai lần. Đồng thời API phải đẹp tới mức dev khác tin tưởng cả business của họ vào đó.

Cách họ giải
  • Idempotency-Key: client gửi một key duy nhất cho mỗi thao tác; Stripe lưu kết quả theo key. Request trùng key → trả lại kết quả cũ, không thực thi lại. Đây là chuẩn vàng cho API tài chính.
  • API versioning chặt chẽ: thay đổi không phá vỡ client cũ — mỗi account "ghim" một version.
  • Thiết kế lỗi rõ ràng, tài liệu xuất sắc, contract ổn định nhiều năm.
Bài học: Ở hệ thống có hệ quả tiền bạc, idempotency không phải tuỳ chọn. Và chất lượng API (rõ ràng, ổn định, versioned) là một phần của kiến trúc — không phải chuyện "code cho xong". Bạn sẽ áp trực tiếp vào dự án Payment #6.
WhatsApp — đội tí hon, quy mô khổng lồ vài chục kỹ sư, hàng trăm triệu user▶

🛠️ Dự án thực hành để "thấm" kiến thức

Lý thuyết chỉ thành kỹ năng khi bạn tự xây. Mỗi dự án gắn với một giai đoạn, đi từ dễ tới khó. Không cần làm production-grade — mục tiêu là tự tay chạm vào vấn đề mà lý thuyết mô tả.

Cách làm dự án cho ra kết quả

Với mỗi dự án: (1) viết một mini-RFC trước khi code — mục tiêu, NFR, sơ đồ C4, lựa chọn & trade-off; (2) build phần lõi (không cần đẹp); (3) tự ép hệ thống gãy — bắn tải, tắt một node, ngắt mạng — rồi quan sát & ghi lại bài học. Chính bước (3) dạy bạn nhiều nhất. Mỗi dự án xong → bỏ vào portfolio kèm RFC + sơ đồ.

Đưa vào queue → API trả "đã đăng" ngay (nhanh), worker xử lý fan-out nền. Queue cũng hấp thụ burst (lúc nhiều người đăng cùng lúc) và cho retry nếu lỗi. Đây chính là lý do dùng async.

Luồng GHI (đăng tweet): Client → Gateway → Tweet Service → ghi DB (+ upload media lên object storage) → bắn event vào Queue → worker fan-out → cập nhật timeline cache (Redis) của các follower. API trả về ngay sau khi ghi DB, không chờ fan-out.

Luồng ĐỌC (xem timeline): Client → Gateway → Timeline Service → đọc list tweet id dựng sẵn từ Redis → lấy nội dung tweet (cache/DB) → media qua CDN → trả về. Toàn bộ đường nóng tránh tính toán nặng.

👉 Mấu chốt tư duy: read-heavy → đẩy chi phí sang lúc GHI (dựng sẵn timeline khi đăng) để lúc ĐỌC chỉ việc tra cứu. Đó là toàn bộ lý do có Queue + Redis ở đây.

Tính timeline lúc đọc (lấy tweet của hàng nghìn người mình follow rồi trộn) thì quá chậm cho 17k+ reads/s. Hai chiến lược:

Giải pháp thực tế (Twitter dùng): HYBRID — user thường dùng fan-out on write; "celebrity" triệu follower dùng fan-out on read để tránh ghi nổ.

Lời giải hybrid: Khi đăng tweet, push vào timeline cache của follower (write). Nhưng với người có > N follower (celebrity), không push; thay vào đó lúc follower mở feed mới kéo tweet của celebrity vào trộn (read). Tránh được "1 tweet = chục triệu lần ghi".

5Trade-off, bottleneck & scale

  • Consistency: chọn AP cho timeline — chậm vài giây chấp nhận được, đổi lấy availability & tốc độ.
  • Bottleneck chính: fan-out cho celebrity (giải bằng hybrid); cache là tuyến phòng thủ đọc (giải bằng Redis + sharding cache).
  • Shard tweet theo user_id hoặc tweet_id; dùng ID có nhúng timestamp (Snowflake) để sắp theo thời gian không cần node trung tâm.
  • Media: object storage + CDN, DB chỉ giữ URL — đã bàn ở GĐ2.
  • Mở rộng tiếp: thêm read replica khi đọc tăng; thêm shard khi ghi/dữ liệu tăng; tách service theo bounded context khi team lớn.
🎯 Ghi nhớ: Mạch Yêu cầu/NFR → Estimation → High-level → Deep dive → Trade-off áp được cho mọi bài. Đáp án không cố định; điều giám khảo (và đời thực) đánh giá là chất lượng lập luận ở bước 4–5.

Chuyên sâu · Messaging

📨 RabbitMQ vs Kafka — hiểu tận gốc & thực tế dùng

Hai cái không phải đối thủ thay thế nhau mà là hai loại công cụ khác bản chất. Hiểu đúng "linh hồn" của mỗi cái → biết chọn đúng và dùng đúng.

① Bản chất: hai triết lý khác nhau hoàn toàn · đọc cái này trước▶

Nếu chỉ nhớ một điều, nhớ điều này:

RabbitMQ = "Message Broker thông minh". Nó định tuyến message tới đúng hàng đợi, đẩy cho consumer, và xoá message sau khi được xử lý xong (ack). Giống bưu điện: thư được chuyển tới đúng hòm, người nhận lấy ra thì thư biến mất.

Kafka = "Distributed Commit Log". Nó ghi sự kiện vào một cuốn nhật ký bền vững theo thứ tự và giữ lại (mặc định 7 ngày, có thể vĩnh viễn). Consumer tự đọc và tự nhớ "đã đọc tới đâu" (offset). Giống một cuốn sổ cái: ghi vào không xoá; nhiều người đọc độc lập, ai muốn đọc lại từ đầu cũng được.

② RabbitMQ chuyên sâu — Exchange, Queue, Ack · broker định tuyến▶
Kiến trúc & đường đi của một message

Điểm mấu chốt: producer không gửi thẳng vào queue. Nó gửi vào Exchange, và Exchange dựa trên routing key + binding để quyết định đẩy vào (các) queue nào.

Producer → Exchange → (binding theo routing key) → các Queue → Consumer ack từng message. Routing nằm ở broker — đây là sức mạnh của RabbitMQ.
Các loại Exchange (cốt lõi)
  • direct: routing key khớp chính xác → đẩy vào queue bound với key đó.
  • topic: khớp theo mẫu với * (một từ) và # (nhiều từ), vd order.*.vn. Linh hoạt nhất.
  • fanout: bỏ qua routing key, broadcast tới mọi queue bound vào → pub/sub.
  • headers: route theo header thay vì routing key (ít dùng).
Đảm bảo tin cậy (phải nắm)
  • Acknowledgement (ack): consumer xử lý xong gửi ack → broker mới xoá. Consumer chết khi chưa ack → message được redeliver. Luôn dùng manual ack (đừng auto-ack) để không mất việc giữa chừng.
  • Prefetch (QoS): giới hạn số message chưa-ack mỗi consumer giữ. Đặt thấp → chia việc công bằng (consumer nhanh nhận nhiều hơn), tránh một consumer ôm hết.
  • Durability: để không mất khi broker restart cần đủ 3 thứ: durable queue + persistent message + publisher confirms (broker xác nhận đã nhận).
  • Dead Letter Exchange (DLX): message bị reject/hết TTL/quá maxlength → chuyển sang DLX → DLQ để xử lý sau. Tránh kẹt hàng đợi.
  • Priority queue, message TTL: hỗ trợ sẵn — RabbitMQ giỏi các tính năng "per-message" tinh tế này.
  • HA: Quorum Queues (dựa trên Raft, replicated) là chuẩn hiện nay cho độ sẵn sàng cao (thay cho mirrored queue cũ).
Giới hạn: RabbitMQ không thiết kế để giữ message lâu / replay. Queue tồn đọng quá nhiều (millions) làm chậm broker. Throughput cao nhưng không bằng Kafka ở mức hàng triệu msg/s. Ordering chỉ đảm bảo trong một queue với một consumer.
③ Kafka chuyên sâu — Partition, Offset, Consumer Group · commit log phân tán▶
Kiến trúc cốt lõi

Một Topic được chia thành nhiều Partition. Partition là đơn vị của song song hoá và thứ tự. Mỗi partition là một log nối tiếp, mỗi message có một offset (số thứ tự) tăng dần.

Topic chia 3 partition. Group A (1 consumer/partition) để xử lý song song. Group B đọc độc lập cùng dữ liệu với offset riêng — đây là lý do nhiều hệ tiêu thụ cùng luồng mà không ảnh hưởng nhau.
Những điểm phải nắm
  • Ordering chỉ trong một partition. Muốn các message liên quan giữ đúng thứ tự (vd theo userId) → dùng cùng key → Kafka hash key về cùng partition. Không có thứ tự toàn cục giữa các partition.
  • Consumer Group: mỗi partition được đọc bởi đúng một consumer trong group → song song tối đa = số partition. Thêm consumer hơn số partition thì dư thừa. Group khác đọc hoàn toàn độc lập (mỗi group có offset riêng) → fan-out tự nhiên.
  • Rebalance: consumer vào/ra group → Kafka chia lại partition. Cần xử lý cho mượt (tránh rebalance liên tục).
  • Replication & độ bền: mỗi partition có replica (leader + follower). acks=all + min.insync.replicas → ghi chỉ thành công khi đủ replica đồng bộ (ISR) nhận → không mất dữ liệu khi leader chết. acks=1 nhanh hơn nhưng rủi ro mất nếu leader chết trước khi sao chép.
  • Retention & Replay: giữ message theo thời gian/dung lượng. Consumer mới có thể đọc lại từ offset 0 → tái dựng state, thêm hệ tiêu thụ mới mà không mất dữ liệu lịch sử.
  • Log Compaction: chế độ giữ bản mới nhất theo key (thay vì theo thời gian) → hợp để lưu "trạng thái hiện tại" (vd snapshot config, CDC).
  • Vì sao throughput khủng: ghi tuần tự xuống đĩa (nhanh như RAM với nhiều ổ), zero-copy khi gửi, batch message, tận dụng page cache OS. Đạt hàng triệu msg/s.
  • Delivery: mặc định at-least-once (có thể trùng → consumer phải idempotent). Exactly-once khả thi trong Kafka qua idempotent producer + transactions (read-process-write), nhưng phức tạp & chậm hơn.
Giới hạn: không có routing thông minh như exchange (consumer tự lọc). Không hợp cho hàng đợi tác vụ cần per-message ack/priority tinh tế. Vận hành nặng hơn (ZooKeeper/KRaft, partition, rebalance). Latency thường cao hơn RabbitMQ ở tải thấp (vì thiết kế cho throughput).

⑤ Thực tế công ty dùng như thế nào · kịch bản & kiến trúc thật▶
RabbitMQ — dùng thực tế cho gì
  • Task / Job queue (phổ biến nhất): web app đẩy việc nền vào queue, worker xử lý. Vd: gửi email/SMS, xuất PDF/báo cáo, resize ảnh, gọi API bên thứ ba. Stack kinh điển: Celery + RabbitMQ (Python), Sidekiq-style, Spring AMQP (Java).
  • Phân phối công việc có routing: định tuyến loại việc tới nhóm worker phù hợp (topic exchange), ưu tiên việc khẩn (priority queue).
  • RPC bất đồng bộ / request-reply giữa các service khi cần phản hồi nhưng không muốn chặn.
  • Order/workflow processing trong hệ vừa & nhỏ: mỗi bước một queue, ack đảm bảo không mất.

Đặc trưng: số message vừa phải, cần xử lý đáng tin từng việc, không cần lưu lịch sử. Lượng lớn ứng dụng web "bình thường" dùng RabbitMQ cho background job là quá đủ.

Kafka — dùng thực tế cho gì
  • Event backbone / "hệ thần kinh trung ương": mọi sự kiện quan trọng (user signup, order placed, payment...) được publish lên Kafka; nhiều service & hệ phân tích cùng subscribe. Đây là cách LinkedIn (nơi sinh ra Kafka), Uber, Netflix dùng ở quy mô hàng nghìn tỉ message.
  • Log & metrics aggregation: gom log/metric từ hàng nghìn service về một luồng để xử lý/lưu trữ.
  • Clickstream & realtime analytics: nuôi stream processor (Flink/Kafka Streams) cho dashboard real-time (xem bài Realtime Analytics #13).
  • Change Data Capture (CDC): Debezium đọc WAL của DB → đẩy lên Kafka → đồng bộ sang search index, cache, data warehouse, service khác (xem bài CDC ở GĐ3).
  • Event Sourcing & data pipeline: Kafka làm log nguồn sự thật; nuôi data lake/warehouse cho cả batch lẫn stream.

Đặc trưng: lượng cực lớn, nhiều consumer độc lập, cần tua lại, dữ liệu là "dòng sự kiện liên tục".

Quan trọng — thực tế chúng cùng tồn tại: Nhiều công ty dùng cả hai: Kafka làm event bus (ghi lại mọi sự kiện cho phân tích & nhiều service), RabbitMQ làm task queue (giao việc nền cho worker). Vd luồng thật: "Order placed" → publish lên Kafka (để analytics, đề xuất, kho dữ liệu cùng nghe) → đồng thời một service đẩy việc "gửi email xác nhận" vào RabbitMQ cho worker email xử lý. ⑥ Cạm bẫy & best practice (kinh nghiệm production) · tránh trả giá▶
Chung cho cả hai
  • Consumer phải idempotent: cả hai mặc định at-least-once → message có thể tới 2 lần (do retry/rebalance). Xử lý phải chịu được trùng (dùng idempotency key / dedup).
  • Luôn có chiến lược cho message lỗi: RabbitMQ → DLX/DLQ; Kafka → topic lỗi riêng + retry topic. Đừng để một message độc làm kẹt cả luồng ("poison message").
  • Theo dõi tồn đọng: RabbitMQ → queue depth; Kafka → consumer lag. Lag/queue tăng dần = consumer không kịp (xem mục Monitoring).
RabbitMQ
  • Dùng manual ack, đừng auto-ack (auto-ack mất việc nếu consumer chết giữa chừng).
  • Đặt prefetch hợp lý (vd 10–50) để chia việc công bằng; prefetch quá cao → một consumer ôm hết, quá thấp → kém throughput.
  • Đừng để queue tồn đọng triệu message (broker chậm). Queue là để chảy qua, không phải để chứa lâu.
  • Bật durable + persistent + publisher confirms cho dữ liệu quan trọng; dùng Quorum Queue cho HA.
Kafka
  • Chọn số partition cẩn thận: nó giới hạn mức song song của consumer; tăng partition sau làm xáo trộn thứ tự theo key. Ước lượng dư một chút từ đầu.
  • Chọn key đúng để vừa giữ ordering cần thiết vừa phân bố đều (tránh hot partition — vd key = userId của celebrity).
  • Đặt acks=all + min.insync.replicas≥2 cho dữ liệu không được mất; hiểu mình đang đánh đổi gì với acks=1.
  • Xử lý rebalance mượt (commit offset đúng lúc); tránh consumer xử lý quá lâu gây rebalance liên tục.
  • Đừng dùng Kafka như một database để query — nó là log; cần query thì đổ vào store phù hợp.

🗄️ Database chuyên sâu — mỗi loại giải bài toán gì

Không có DB "tốt nhất" — chỉ có DB hợp access pattern. Phần này mổ xẻ các DB hàng đầu hiện nay: bản chất, giải bài toán nào, vì sao nhanh ở việc đó, và khi nào KHÔNG dùng.

Tư duy chọn DB (đọc trước)

Hỏi 4 câu: (1) Truy cập theo kiểu gì? (theo key? query phức tạp? full-text? quan hệ? tổng hợp số liệu?) (2) Đọc nhiều hay ghi nhiều? (3) Cần transaction/consistency mạnh tới đâu? (4) Quy mô dữ liệu & tăng trưởng? → Câu trả lời quyết định loại DB. SQL (Postgres) là mặc định; chỉ đổi khi có lý do cụ thể.

PostgreSQL — "the default" đa năng · OLTP quan hệ▶
Giải bài toán gì

Phần lớn ứng dụng "bình thường": dữ liệu có quan hệ, cần transaction ACID, query linh hoạt (JOIN, filter, aggregate). Là điểm khởi đầu đúng cho ~90% dự án.

Vì sao mạnh
  • ACID đầy đủ, MVCC (đọc không chặn ghi), isolation level rõ ràng.
  • Cực giàu tính năng: JSONB (làm document store nhẹ), full-text search, GIS (PostGIS), extension (TimescaleDB, pgvector cho AI embedding), window function, CTE.
  • B-Tree index → query & range tốt; tối ưu query mạnh.
Khi nào tới hạn

Một node Postgres đi rất xa (read replica + cache + sharding thủ công kéo tới hàng chục nghìn TPS). Tới hạn khi: ghi vượt 1 master, dữ liệu quá lớn cho 1 máy, hoặc cần analytics quét tỉ dòng (đó là lúc cân nhắc Cassandra/ClickHouse/NewSQL).

Chốt: Đừng vội bỏ Postgres. "Just use Postgres" đúng đáng ngạc nhiên ở quy mô lớn (Instagram, Notion, Figma đều dựa nền Postgres). Chỉ đổi/bổ sung khi có bằng chứng nó không kham.
Cassandra / ScyllaDB — ghi khổng lồ, không bao giờ chết · wide-column, masterless▶
Giải bài toán gì

Khi bạn cần ghi cực lớn liên tục (hàng trăm nghìn–triệu writes/s), không được downtime, và trải nhiều datacenter/vùng địa lý. Ví dụ kinh điển: lưu tin nhắn chat, sự kiện/log, time-series, feed, dữ liệu IoT.

Cách hoạt động — vì sao nó làm được điều đó
Cassandra không có master (peer-to-peer) → không có SPOF, ghi vào node nào cũng được. Consistent hashing phân bố dữ liệu, quorum cho consistency tuỳ chỉnh, LSM-Tree cho ghi nhanh. Scale tuyến tính bằng cách thêm node.
Đặc điểm cốt lõi cần nắm
  • Masterless / AP: mọi node ngang hàng, chịu được mất node & chia cắt mạng → availability cực cao. Theo CAP thiên về AP (tunable: chọn consistency level per-query).
  • Tunable consistency: bạn chọn ONE / QUORUM / ALL cho mỗi đọc/ghi → đánh đổi tốc độ vs đúng đắn theo từng truy vấn.
  • Query-driven data modeling: KHÔNG thiết kế bảng theo dữ liệu rồi mới query (như SQL). Phải biết trước query rồi thiết kế bảng phục vụ đúng query đó (thường nhân bản dữ liệu — denormalize). Không có JOIN.
  • Partition key quyết định tất cả: chọn sai → hot partition hoặc query phải quét nhiều node.
Điểm yếu / khi nào KHÔNG dùng
  • Không hợp query ad-hoc, không JOIN, không transaction đa bảng. Đổi yêu cầu query sau này rất khổ (phải tạo bảng mới).
  • Tombstone (xoá tạo marker) gây chậm nếu xoá nhiều. Đọc có thể chậm hơn ghi.
  • Vận hành phức tạp (repair, compaction tuning).
Dùng thực tế & ScyllaDB

Netflix, Instagram, Apple, Uber dùng Cassandra cho dữ liệu quy mô khổng lồ. ScyllaDB = viết lại bằng C++ (kiến trúc shard-per-core, không GC pause) → tương thích Cassandra nhưng latency p99 ổn định hơn & ít máy hơn. ClickHouse — phân tích tỉ dòng trong mili-giây · columnar OLAP▶
Giải bài toán gì

Analytics / OLAP: tổng hợp (SUM, COUNT, AVG, GROUP BY) trên hàng tỉ dòng gần như tức thì — cho dashboard real-time, phân tích log/event/clickstream, BI. Đây là loại truy vấn mà Postgres/MySQL "chết" vì phải quét toàn bảng.

Cách hoạt động — vì sao nhanh kinh khủng
Row store lưu theo dòng → tính tổng 1 cột vẫn phải đọc mọi cột. Columnar lưu theo cột → chỉ đọc đúng cột cần + nén cực mạnh (giá trị giống nhau cạnh nhau) + xử lý vector hoá → nhanh hơn hàng trăm lần cho aggregation.
Ba lý do ClickHouse nhanh
  • Columnar storage: chỉ đọc cột cần cho query (analytics thường chỉ dùng vài cột trên bảng rộng).
  • Nén cực mạnh: dữ liệu cùng cột giống nhau → nén 10x+ → đọc ít byte từ đĩa hơn.
  • Vectorized execution: xử lý cả khối giá trị một lúc (SIMD), tận dụng CPU cache → throughput khủng.
Điểm yếu / khi nào KHÔNG dùng
  • Không phải OLTP: cập nhật/xoá 1 dòng là "mutation" bất đồng bộ & tốn kém — đừng dùng làm DB ứng dụng chính.
  • Không có transaction đầy đủ, không hợp point-lookup tần suất cao (lấy 1 record theo id liên tục).
  • JOIN lớn yếu hơn DB chuyên dụng; thường denormalize trước.
Dùng thực tế

Cloudflare (phân tích traffic khổng lồ), Uber, hàng loạt nền tảng observability/BI. Thường nằm ở cuối pipeline: dữ liệu thô → Kafka → ClickHouse để truy vấn analytics (xem bài Realtime Analytics #13). Cặp đôi điển hình: Postgres cho OLTP + ClickHouse cho OLAP.

Phân biệt cốt lõi: OLTP (Postgres/Cassandra) = nhiều giao dịch nhỏ, đọc/ghi theo record. OLAP (ClickHouse) = ít truy vấn nhưng mỗi cái quét/tổng hợp khối dữ liệu khổng lồ. Đừng nhầm vai trò của chúng.
Redis · MongoDB · DynamoDB · Elasticsearch · Neo4j · Time-series · các chuyên dụng còn lại▶
Redis — in-memory, sub-millisecond

Giải: cần truy cập cực nhanh (RAM). Dùng cho cache, session, rate limiter, leaderboard (sorted set), distributed lock, pub/sub, hàng đợi nhẹ (Streams), geo. Không dùng làm nguồn sự thật chính cho dữ liệu lớn bền (dù có persistence) — RAM đắt & giới hạn.

MongoDB — document, schema linh hoạt

Giải: dữ liệu dạng JSON lồng nhau, schema thay đổi liên tục (giai đoạn đầu sản phẩm), không cần JOIN nhiều. Mạnh: dev nhanh, sharding sẵn. Yếu: quan hệ phức tạp & transaction xuyên nhiều document (có hỗ trợ nhưng không phải thế mạnh). Lưu ý: Postgres + JSONB thường thay thế được MongoDB cho nhiều case.

DynamoDB — KV managed, serverless

Giải: cần latency ổn định (single-digit ms) ở mọi quy mô mà không muốn vận hành DB. AWS lo hết scaling. Mạnh: auto-scale, bền, dễ. Yếu: phải thiết kế key & access pattern từ đầu (giống Cassandra), query linh hoạt kém, chi phí dễ phình nếu thiết kế sai. Dùng nhiều cho serverless trên AWS.

Elasticsearch — full-text search & log

Giải: tìm kiếm văn bản (inverted index), gợi ý, lọc/facet, và phân tích log (bộ ELK: Elasticsearch + Logstash + Kibana). Yếu: không phải nguồn sự thật, không transaction; thường đồng bộ từ DB chính (qua CDC). Đừng dùng làm primary DB.

Neo4j — graph

Giải: khi quan hệ là trung tâm & cần đi sâu nhiều cấp (bạn-của-bạn, đề xuất, phát hiện gian lận, knowledge graph). Truy vấn graph traversal cực nhanh so với JOIN đệ quy trong SQL. Yếu: không hợp dữ liệu bảng phẳng đơn giản.

Time-series (TimescaleDB / InfluxDB)

Giải: dữ liệu theo thời gian khối lượng lớn — metrics, IoT sensor, giá cổ phiếu. Tối ưu ghi theo thời gian, nén, downsampling, truy vấn theo cửa sổ thời gian. TimescaleDB = extension trên Postgres (giữ SQL).

NewSQL (CockroachDB / Spanner / YugabyteDB)

Giải: muốn SQL + ACID mạnh NHƯNG cũng cần scale ngang & đa vùng như NoSQL — "best of both". Dùng khi Postgres một node không đủ mà vẫn cần transaction phân tán (fintech toàn cầu). Đánh đổi: phức tạp & latency cao hơn; đừng dùng khi 1 Postgres còn đủ.

📊 Monitoring & Cảnh báo — theo dõi gì, ngưỡng bao nhiêu

Chỉ số cần theo dõi ở từng tầng, ngưỡng tham khảo, vì sao ngưỡng đó, và làm gì khi vượt. Đây là kiến thức vận hành mà Architect phải định nghĩa cho hệ thống.

3 nguyên tắc vàng trước khi xem ngưỡng

1. Cảnh báo theo "triệu chứng", không theo "nguyên nhân". Page người trực khi user bị ảnh hưởng (latency cao, lỗi nhiều), không phải mỗi khi CPU 85%. CPU cao mà user vẫn ổn thì không cần dựng người dậy lúc 3h sáng.

2. Luôn dùng percentile (p95/p99), không dùng trung bình. Trung bình che giấu nhóm user bị chậm. "Latency trung bình 50ms" có thể vẫn có 1% user chờ 5 giây.

3. Ngưỡng tốt nhất đến từ baseline của chính bạn + biên an toàn, không phải con số tuyệt đối. Các số dưới đây là điểm khởi đầu — đo hệ thống lúc khoẻ mạnh, rồi đặt ngưỡng cao hơn mức bình thường một khoảng hợp lý.

🚨 Cảnh báo dựa trên SLO & Error Budget (cách "best")

Đặt ngưỡng tĩnh cho mọi thứ dễ gây alert fatigue (cảnh báo nhiều quá → người trực phớt lờ → bỏ lỡ cái thật). Cách hiện đại:

  • Định nghĩa SLO (vd 99.9% request < 300ms trong 30 ngày) → suy ra error budget (0.1%).
  • Burn-rate alerting: cảnh báo theo tốc độ tiêu error budget. Tiêu nhanh (sẽ cháy budget trong 1 giờ) → page khẩn. Tiêu chậm (cháy trong nhiều ngày) → ticket thường. Đa cửa sổ (5 phút + 1 giờ) để vừa nhạy vừa ít báo nhầm.
  • Phân tầng nghiêm trọng: Page (dựng người dậy — chỉ khi user bị ảnh hưởng & cần hành động ngay) vs Ticket (xem trong giờ làm) vs Dashboard (chỉ quan sát).
  • Mỗi alert phải actionable: nếu nhận alert mà không biết làm gì → alert đó sai, nên bỏ hoặc sửa.
Bộ công cụ phổ biến: Prometheus (thu metrics) + Grafana (dashboard) + Alertmanager (cảnh báo); OpenTelemetry (chuẩn cho tracing/metrics/logs); Loki/ELK (logs); Jaeger/Tempo (tracing); PagerDuty/Opsgenie (on-call). Cloud: CloudWatch, Datadog, New Relic.
Nhắc lại điều quan trọng nhất: Mọi ngưỡng ở trên là điểm khởi đầu. Hệ thống của bạn có baseline riêng — một dịch vụ batch chạy CPU 95% là bình thường, một API ở mức đó là sắp gãy. Đo lúc khoẻ → đặt ngưỡng tương đối → cảnh báo trên triệu chứng user-facing → tinh chỉnh để giảm báo nhầm. Đó mới là monitoring "best".

Nền tảng & Data

Q. Khi nào horizontal scaling khó áp dụng, và điều kiện để nó dễ?
Khó khi service stateful (giữ session trong RAM) — request phải về đúng server. Điều kiện để dễ: service stateless, đẩy state ra Redis/DB dùng chung hoặc token tự chứa. Khi đó thêm/bớt máy tuỳ ý.
Q. CAP: khi mạng đứt, hệ ngân hàng nên chọn CP hay AP? Còn feed mạng xã hội?
Ngân hàng → CP (thà chặn/từ chối còn hơn sai số dư). Feed → AP (thà hiện dữ liệu cũ vài giây còn hơn không hiện). CAP chỉ áp lúc có partition; bình thường có cả C lẫn A.
Q. Vì sao Cassandra ghi rất nhanh còn Postgres truy vấn linh hoạt hơn?
Cassandra dùng LSM-Tree (ghi nối tiếp vào bộ nhớ rồi flush — tối ưu ghi). Postgres dùng B-Tree (tối ưu đọc & range query). Đây là đánh đổi read-heavy vs write-heavy.
Q. Cache stampede là gì và chống thế nào?
Một key hot hết hạn cùng lúc → hàng nghìn request đập thẳng DB. Chống: lock/single-flight (chỉ 1 request tính lại), làm mới chủ động trước khi hết hạn, hoặc randomize TTL.
Q. Consistent hashing giải quyết vấn đề gì so với hash % N?
hash % N: thêm/bớt 1 node đổi N → gần như mọi key phải di chuyển. Consistent hashing: chỉ key trên một cung nhỏ phải di chuyển. Virtual node giúp phân bố đều.

Distributed & Architecture

Q. Vì sao mọi thao tác có side-effect tiền bạc phải idempotent?
Vì trong hệ phân tán request chắc chắn bị gửi lại (timeout/retry). Không idempotent → trừ tiền/tạo đơn 2 lần. Giải: client gửi idempotency key, server thấy key trùng thì trả kết quả cũ.
Q. Tại sao thường tránh 2PC và chọn Saga trong microservices?
2PC khoá tài nguyên lâu, chậm, và coordinator chết là kẹt toàn bộ. Saga chia thành transaction cục bộ + compensation để hoàn tác → đổi atomicity tức thời lấy eventual consistency, phù hợp quy mô lớn.
Q. "Distributed monolith" là gì và vì sao tệ?
Là microservices nhưng ranh giới vẽ sai → các service vẫn dính chặt, đổi cái này phải deploy cái kia, nhưng giờ phải gọi qua mạng. Tệ hơn cả monolith. Nguyên nhân: tách quá sớm khi chưa hiểu domain → nên monolith first + chia theo bounded context (DDD).
Q. Phân biệt API Gateway và Service Mesh?
API Gateway = cổng vào cho client (north-south): auth, rate limit, routing. Service Mesh = lo giao thông giữa các service (east-west): mTLS, retry, circuit breaker qua sidecar, không cần sửa code service.

Reliability & tư duy Architect

Q. Vì sao thiếu Timeout dễ gây cascading failure?
Không có timeout → caller chờ mãi service chậm → cạn thread/connection pool → caller cũng treo → lan ngược toàn hệ thống. Timeout + circuit breaker chặn dây chuyền này.
Q. Error budget dùng để làm gì?
Là phần được phép lỗi (vd 0.1% nếu SLO 99.9%). Còn budget → cứ ship feature; cháy budget → dừng feature, dồn sửa reliability. Cách dùng số liệu để cân bằng tốc độ vs ổn định khách quan.
Q. Câu hỏi đầu tiên nên hỏi khi nhận một bài thiết kế là gì?
Làm rõ NFR & scope: bao nhiêu user/QPS? đọc nhiều hay ghi nhiều? consistency cần mức nào? latency mục tiêu? ngân sách? Bỏ qua bước này = thiết kế sai quy mô (lỗi lớn nhất).
Q. Vì sao "dữ liệu sống lâu hơn code" ảnh hưởng cách phân bổ thời gian thiết kế?
API/code refactor trong một sprint, nhưng schema/shard key sai có thể buộc migration hàng tháng. Nên dồn thời gian & cẩn trọng nhất cho thiết kế dữ liệu và ranh giới ownership dữ liệu.
Q. Một câu tóm tắt: điều gì phân biệt Architect giỏi?
Không phải biết nhiều pattern nhất, mà là kỷ luật chọn giải pháp đơn giản nhất đủ dùng, thiết kế cho lúc hỏng, nêu rõ trade-off và ghi lại lý do. Trưởng thành đo bằng số thứ bạn không thêm vào.
